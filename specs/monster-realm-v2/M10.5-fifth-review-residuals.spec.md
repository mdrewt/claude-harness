## 1. Problem / intent

The fifth multi-lens review of the delivered spine (M0–M9 + M10a) found the architecture **sound and clean** on the load-bearing lenses — server-authority/reject-not-clamp, privacy split-tables (structurally enforced: the SpacetimeDB codegen emits no client accessors for private tables), determinism, SSOT, illegal-states/parse-don't-validate, additive-schema safety, and the production stub-sweep all hold. It surfaced a small, **high-ROI, low-risk** residual set worth clearing before the durable docs drift further and before M10 adds evolved/derived forms:

1. **One latent reducer-panic class.** `pick_best_skill` ends in `.expect("…at least one known skill…")`; the player side is guarded but the **enemy/wild** side is not, and `validate_content` never enforces that a species' `learnable_skill_ids` is non-empty. Not reachable on today's content (all six shipped species have non-empty, resolving movesets) but the invariant the `.expect` relies on is unenforced — exactly as M10 begins minting derived/evolved species. Per ADR-0049 a core panic is only legitimate when `validate_content` actually guarantees it.
2. **Durable-doc drift frozen at the M8.9 close.** `ARCHITECTURE.md` (the design record + the *canonical `touches:` vocabulary* the autonomous runner reasons about) and the `docs/adr/` index never advanced past M8.9: the module map omits `inventory.rs`/`raising.rs` and mis-files `grant_item`/`consume_one`; the content-registry, decisions list, and subsystem sections omit M9 raising + M10a evolution/fusion; the ADR index "Next free number" still points at `0060` (which exists -> next author collides). A wrong `touches:` map can make the runner mis-judge two slices as disjoint.
3. **Cheap mechanical-gate hardening.** Three small gaps where a future regression could slip a gate: no `allowOnly:false`/`forbidOnly` (a stray `.only` silently narrows the client gate), `evals/run.mjs` aborts the loop on the first throwing eval (still red, but hides later evals), and `flushBatch` has no per-listener isolation (a throw starves sibling batch listeners — a tracked M8.8e residual).

Success = the latent panic made impossible by construction with a biting gate; the doc set matches the code; the three mechanical guards in place; `just ci` green-and-meaningful. **No player-facing behavior changes.** Genuine policy/cost trade-offs (server-module mutation coverage, a Rust coverage ratchet, the silently-skipped M8.95 knowledge bundle, the position-divergence render snap) are recorded in §5 as **decisions for Drew**, not auto-scheduled here.

## 3. Acceptance criteria (EARS) + located evidence (@ `d873a93`)

### 10.5a — Empty-moveset panic becomes a validated content invariant
Evidence: panic site `game-core/src/combat/ai.rs:55` (`best_id.expect("attacker must have at least one known skill in the skill registry")`); unguarded enemy path `server-module/src/battle.rs:489` (`submit_attack` -> `pick_best_skill(side_b.active_monster(), …)`); player side IS guarded `battle.rs:475-480`; missing invariant `game-core/src/content.rs:550-559` (`validate_content` checks each `learnable_skill_id` exists, never non-empty); empty-list builder `server-module/src/marshal.rs:154-167` (`wild_battle_monster` returns `Ok` with empty `known_skill_ids`). Note the core already guards the empty case at `resolve.rs:374` (recruit-failure path) — this just extends the same defense to the attack path.

- **10.5a-1 (WHERE):** WHERE `validate_content` iterates species, IF a `Species.learnable_skill_ids` is empty, THE rule SHALL return `Err` (a new `ContentError` variant, e.g. `SpeciesHasNoLearnableSkills { species_id }`), so empty movesets are rejected at load/seed, not at battle time.
- **10.5a-2 (WHEN, defense-in-depth):** WHEN `wild_battle_monster` filters `learnable_skill_ids` to seeded skills and the result is empty, THE function SHALL return `Err` (mirroring the existing `stat_defense == 0` guard at `marshal.rs:204-209`), so a content/seed mismatch fails the reducer cleanly with a reject rather than a panic.
- **10.5a-3 (proof-of-teeth):** THE suite SHALL add a game-core gating test that a species with `learnable_skill_ids: []` makes `validate_content` return the new `Err` (RED before 10.5a-1, GREEN after), AND a test that `pick_best_skill` on an empty-moveset attacker is never reached because construction is rejected (or, if kept as a documented invariant, that the guard at the boundary returns `Err`). The teeth MUST bite: reverting either guard turns the test RED.
- **10.5a-4 (no behavior change on valid content):** THE existing battle/encounter tests and all evals SHALL stay GREEN — every shipped species (`content/species/000-core.ron`, `010-derived.ron`) already has a non-empty, resolving moveset, so no valid content is newly rejected.

### 10.5b — ARCHITECTURE.md + README.md reconciled with M9/M10a code
Evidence: module map `ARCHITECTURE.md:118-130` lists 9 modules, omits `inventory.rs`+`raising.rs` (both declared `server-module/src/lib.rs:26,30`) and files `grant_item`/`consume_one` under `taming.rs` though they live at `inventory.rs:47,82` (`pub(crate) fn`, moved in ADR-0059); content-registry table `:160-167` omits `content/evolutions.ron`, `content/fusion.ron`, `content/species/010-derived.ron`; decisions list `:176-187` stops at ADR-0057; no Raising/Evolution subsystem section and `Status` ends at M8.9 (`:419,:516,:522-599`); `README.md:3` says `server/` (the dir is `server-module/`).

- **10.5b-1:** THE `ARCHITECTURE.md` server-module module-map table SHALL add an `inventory.rs` row (owns `grant_item`/`consume_one` — the single item-mutation surface, ADR-0059) and a `raising.rs` row (owns `care`/`train`, ADR-0058/0059), and SHALL remove `grant_item`/`consume_one` from the `taming.rs` row (taming owns `attempt_recruit` + `grant_bait`).
- **10.5b-2:** THE content-registry table SHALL add `evolutions` (`content/evolutions.ron`, single file) and `fusion` (`content/fusion.ron`, single file) rows and note `species/` now holds `000-core.ron` + `010-derived.ron`.
- **10.5b-3:** THE decisions highlight + the "implementation ADRs 0035–00xx" prose SHALL extend through `0060` and reference 0058 (raising EV/care), 0059 (server raising), 0060 (evolution/fusion content shape).
- **10.5b-4:** THE document SHALL gain a Raising subsystem section (M9: pure rules + server `care`/`train`/cooldown/inventory-reconcile + the client raising/inventory view) and an Evolution/Fusion content section (M10a: ADR-0060 content shape + `validate_evolution_fusion`), and the `Status` block SHALL advance through M9/M10a; the forward-looking "M9 raising is future" wording (`:516`) SHALL be corrected.
- **10.5b-5:** `README.md:3` `server/` SHALL become `server-module/`.

### 10.5c — docs/adr/ index + numbering reconciled
Evidence: `docs/adr/README.md:8` ("Implementation ADRs `0001`, `0035`–`0054`" — files 0055–0060 exist), `:13` ("Next free number: `0060`" — 0060 exists, accepted), table ends at 0059 (no 0060 row); `docs/adr/0060-evolution-fusion-content-shape.md:13` self-references "next-free … is 0060"; spec corpus cites ADR-0018/0019 while real are 0058/0059/0060.

### 10.5d — Mechanical gate hardening (no behavior change)
Evidence: `client/package.json:9` (`"test": "vitest run"`, no flags) + `client/vite.config.ts:22` (`test:` block, no `allowOnly`) + `client/playwright.config.ts` (no `forbidOnly`); zero `.only`/`.skip` in client today. `evals/run.mjs:20-26` (no per-eval try/catch; a throw aborts the loop — still non-zero exit, so no false GREEN). `client/src/net/store.ts:244-248` (`flushBatch` iterates listeners with no try/catch; main.ts:233 already documents the hazard; reconcile self-wraps its one known throw at main.ts:113-116).

- **10.5d-1:** THE vitest config SHALL set `allowOnly: false` and Playwright SHALL set `forbidOnly: !!process.env.CI`, so a stray `.only` fails CI instead of silently narrowing the gate. A teeth check (or a comment-documented manual verification) SHALL confirm an added `.only` reds the gate.
- **10.5d-2:** THE `evals/run.mjs` loop SHALL wrap each eval invocation in try/catch, recording a synthetic `pass:false` for a thrower and continuing, while still exiting non-zero if any eval failed — so one broken eval no longer hides the rest.
- **10.5d-3:** THE `store.flushBatch` SHALL invoke each batch listener inside its own try/catch (log + continue), so one throwing listener cannot starve the others (closes the tracked M8.8e residual). Existing batch/reconcile tests stay GREEN.

## 5. Decisions for Drew (NOT auto-scheduled — cost/policy trade-offs)

Each is a real, verified gap whose resolution is a judgment call, not a flat fix. Pick a branch; the runner should not decide these silently.

- **D-mutation (server-module authority).** Nightly mutation is `cargo mutants -p game-core` only (`justfile:47`); the ownership/legality reducers (`attempt_recruit`, `require_owner`, `care`/`train`) live in `server-module` and are proven only by unit tests + static substring-scan evals (which check a guard's *shape*, not its *semantics*). **Choose:** (a) add a nightly `cargo mutants -p server-module` job (cost: nightly runtime; needs to build without a live SpacetimeDB), or (b) record in ADR-0050 that server-module authority rests on unit tests + static scans as an accepted residual.
- **D-coverage (Rust rule core).** The only coverage gate is client vitest at 25% lines; there is no Rust coverage gate for `game-core`/`server-module`. **Choose:** add a `cargo llvm-cov`/tarpaulin ratchet for `game-core`, or document that "coverage" is deliberately client-only. (The older "diluted by gitignored bindings" concern is already resolved — bindings are committed and excluded at `vite.config.ts:42`.)
- **D-M8.95 (silently-skipped milestone). DECIDED (Drew, 2026-07-03):** BUILD M8.95 as its own named slice per `M8.95-knowledge-bundle.spec.md` (OKF-aligned generated schema/reducer projection, drift-gated; ADR-0057 stays on M8.9e/content-dir-glob; the knowledge-bundle corpus ADR is separate). Scheduled after the remaining M12.5 residual slices, or fan-out alongside a disjoint slice per PLAN §9 rules. See M12.5 §2.12.5g-3.
- **D-render-snap (position divergence).** `snapped` is produced only from a local *time* gap (`predictor.ts:219-221`); a large *authoritative position* jump (server correction/warp/respawn, or dropped-update catch-up) glides multiple tiles over one `STEP_MS` instead of snapping (`renderResolver.ts:59-60`). Low-frequency today (no teleport/warp feature). **Choose:** fix now (add a position-divergence snap when the new target is >1 tile from the clock's current target, with a `snapped=false` large-jump test) to pre-empt the glitch before warps land, or defer until a teleport/warp feature exists. *Related low residuals if this is taken up:* the reconcile-internal-drain swallows a real time-gap snap (`predictor.ts:198` discards `snapped` but resets `#lastDrainAt`), and remote sprite pose/position skew ~1.5 steps (`renderResolver.ts:73-79`, cosmetic).
