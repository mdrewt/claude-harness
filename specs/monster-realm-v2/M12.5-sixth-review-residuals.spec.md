## 1. Problem / intent

The sixth review found the spine still sound on the load-bearing lenses (server authority, privacy split-tables, determinism gate, SSOT thinness of reducers, boundary type rigor, dependency hygiene). But the M10b fusion reducer ships a **production-breaking dual-write bug**, the M11c client zone handling has **two user-visible state bugs** (reconnect strand, vanishing idle remotes), and the headline v2 promise "content update without `--delete-data`" (G1, ADR-0006/0037) is **dead code end-to-end**. Plus a set of verified medium/low residuals in netcode feel, battle lifecycle, gate teeth, and docs.

Severity counts (all verified): 1 Critical, 3 High, ~12 Medium, ~14 Low.

## 2. Slices, acceptance criteria (EARS) + located evidence (@ `85c7fd8`)

### 12.5a — CRITICAL: `fuse` offspring `monster_pub` dual-write uses id 0
Evidence: `server-module/src/evolution.rs:355-361` — `pub_from_monster(&offspring_monster)` is computed BEFORE `ctx.db.monster().insert(offspring_monster)` assigns the `auto_inc` id and the insert's return is discarded; `MonsterPub.monster_id` is `#[primary_key]` with NO `auto_inc` (`schema.rs:214-216`), so the pub row lands with `monster_id = 0`. Every other creation path uses the returned row (`movement.rs:104-105`, `taming.rs:136-137`). Consequences: fusion offspring invisible to clients (clients render from `monster_pub`); the SECOND fusion on a database collides on pub PK 0 and aborts — fusion permanently broken after one use; all later dual-writes keyed by the real id miss. Tests pass because the seam `fuse_seam` (`evolution_tests.rs:1499-1504`) pre-allocates the id — the seam diverges from production at exactly the buggy line.

- **12.5a-1:** WHEN `fuse` inserts the offspring, THE reducer SHALL capture `let inserted = ctx.db.monster().insert(offspring_monster)` and build the pub row from `inserted` (mirroring `movement.rs:104-105`).
- **12.5a-2:** THE `fuse_seam` test double SHALL be aligned so the id is assigned at insert time (not pre-allocated), making the seam faithful to production ordering.
- **12.5a-3 (proof-of-teeth):** THE suite SHALL add a gating test / eval tooth asserting the monster↔monster_pub dual-write invariant on the fuse path: after a fuse, no `monster_pub` row has `monster_id = 0` and the offspring's pub row exists with the real id. Reverting 12.5a-1 turns it RED. (Consider extending `monster-dual-write.eval.mjs`, which today misses this ordering bug.)
- Touches: `server-module/src/evolution.rs`, `evolution_tests.rs`, `evals/monster-dual-write.eval.mjs`. Disjoint from M12a (game-core npc/dialogue/quest).

### 12.5b — HIGH: content-sync path repair (G1/ADR-0006 promise is currently dead)
Evidence: `sync_content` rejects every caller (`lib.rs:104-112` guards `ctx.sender != ctx.identity()`; the module identity never calls it — not scheduled, and `sync_content_inner` is invoked only from `init`, `lib.rs:96`); `just publish` (justfile:60-61, no `--delete-data`) therefore can NEVER refresh content on a live DB. No re-derive pass exists: `sync_content_inner` (`content.rs:25-304`) never touches `monster`/`monster_pub`, so a base-stat change leaves stale stored `stat_*` (and possibly `current_hp > stat_hp`). Partial-commit: every failure arm is `log::error!(); return;` from a `()` helper, so late validation failure (e.g. `validate_evolution_fusion` at :267 after encounter writes at :215) COMMITS earlier registry writes with no version stamp. `CONTENT_VERSION` (lib.rs:38-41) is manually bumped with no mechanical coupling to `content/**.ron` changes. `evolves_to` is additionally stale on creation/level-up/care: `monster_from_instance` hardcodes `None` (`marshal.rs:86`), and `compute_evolves_to` is called only in `evolution.rs` — contradicting the `schema.rs:202-204` comment.

- **12.5b-1:** THE `sync_content` reducer SHALL be callable by the database owner (owner-identity check replacing the unreachable module-identity guard, per the project's authz conventions), so `just publish` + `spacetime call sync_content` is a working live-content-update path. (ADR update: amend ADR-0037's "callable on republish" mechanism note.)
- **12.5b-2:** THE `sync_content_inner` SHALL load + validate ALL registries BEFORE writing any row (or propagate `Err` so the reducer transaction rolls back atomically) — a late validation failure SHALL leave the DB unchanged.
- **12.5b-3:** WHEN content is re-seeded with a changed version, THE system SHALL run a re-derive pass over `monster` rows (recompute `derive_stats` + `compute_evolves_to`, dual-write `monster_pub`, clamp `current_hp` to the new `stat_hp`) in the same transaction.
- **12.5b-4:** THE creation paths (`monster_from_instance` callers), the battle level-up write-back, and `care` SHALL recompute `evolves_to` so the stored UI hint matches the `schema.rs` comment (or the comment SHALL be corrected to the narrower truth — implementer's call, but the UI-hint staleness for leveled/bonded monsters should be fixed, not documented away).
- **12.5b-5 (proof-of-teeth):** an eval SHALL couple `game-core/content/**` changes to `CONTENT_VERSION`: hash the content tree into a committed baseline keyed by the version; content change without a bump fails CI. Teeth: doctored fixture must fail.
- **12.5b-6 (DECIDED by Drew, 2026-07-03):** APPROVED as a **nightly job** (not per-PR): publish → edit content → republish WITHOUT `--delete-data` → assert data survives + new content served. **Failure policy (Drew):** any nightly failure is inserted into the milestone slice queue as the NEXT slice to work on and resolve when detected (supervisor picks it up as priority target, same tier as fix-red-master but below it). Implement as sub-slice **12.5b-6-impl** (nightly workflow + the smoke e2e itself).
- Touches: `server-module/src/lib.rs`, `content.rs`, `marshal.rs`, `battle.rs` (write-back), `raising.rs`, `evals/`. Collides with M12b/M12c server slices if they touch content seeding — serialize after M12.

### 12.5c — HIGH: client zone-sync robustness (M11c residuals)
Evidence: (1) Reconnect strand — server `on_disconnect` deletes the character (`lib.rs:114-121`), rejoin spawns at zone 0 (`movement.rs:53-56`), but client `onReconnect` only resets prediction (`main.ts:434-438`); warp detection is edge-triggered in `onUpdate` only (`connection.ts:108-126`), so a player who disconnects in zone 1 is permanently invisible (render filter `renderResolver.ts:52` skips own row), camera pinned at origin (`main.ts:497-499`), prediction on the wrong map — unrecoverable without reload. (2) `store.resetCharacters()` in the warp handler (`main.ts:447`) clears ALL characters; the subscription is global one-shot and `movement_tick` skips idle rows (`movement.rs:186-192`), so idle remotes never re-appear until they move. (3) Warp-handler failure atomicity: `renderer.setMap` → `TileMap.fromRaw` can throw AFTER `set_active_zone`/reset/`rawMap` mutations (`main.ts:440-453`), leaving split-brain despite the comment claiming atomicity. (4) A wasm `apply_move` throw kills the rAF loop permanently (`main.ts:459-503` — no try/catch, rAF re-arm is the last statement) and `predictor.reconcile` (`main.ts:128`) is outside the listener's try block (extends 10.5d-3).

- **12.5c-1:** ZONE SYNC SHALL BE STATE-BASED: in the batch-applied reconcile listener, WHEN `own.row.zoneId !== rawMap.zone_id`, THE client SHALL run the full zone-switch body (zone_map → parse/validate FIRST → set_active_zone → setMap → resetPredictionState). This one change fixes reconnect-strand AND subsumes the edge-triggered warp path (keep `onOwnWarp` or delete it — implementer's call, covered by tests either way).
- **12.5c-2:** THE warp/zone-switch handler SHALL NOT clear remote characters (rely on the render-side zone filter), OR SHALL re-hydrate the store from the SDK cache (`conn.db.character.iter()`) after clearing; idle remotes in the destination zone SHALL be visible immediately after a warp (unit test with an idle remote fixture).
- **12.5c-3:** THE zone-switch body SHALL perform all fallible parsing (`TileMap.fromRaw`) BEFORE mutating any state, making the "consistent failure" comment true.
- **12.5c-4:** THE frame loop SHALL contain wasm/predictor throws (try/catch with rAF re-arm in `finally`) and the reconcile call SHALL be inside the listener's containment, so one throw cannot permanently halt rendering or starve sibling listeners.
- **12.5c-5 (proof-of-teeth):** an e2e or integration test SHALL cover disconnect-in-zone-1 → reconnect → own character visible at zone 0 spawn, camera following, prediction on zone 0 map.
- Touches: `client/src/main.ts`, `net/connection.ts`, `net/store.ts`, `client/e2e/`. Disjoint from M12a (game-core); collides with M12's client slice if any — serialize per touches.

### 12.5d — MEDIUM: netcode smoothness residuals (ADR-0013 feel bugs in fresh M11c code)
Evidence: (1) `INTERP_DELAY_STEPS = 1.5` (`render/config.ts:16`) needs ≥300 ms of history but the store keeps 2 snapshots (`net/store.ts:185-189`); under steady 200 ms cadence remotes hold ~100 ms → ramp to 0.5 → half-tile jump every cycle (`interpolation.ts:40-48` clamp). (2) No snap-on-teleport for remotes: `prev` chains across zone changes/large deltas; masked today only because both warps are (5,5)→(5,5). (3) Snap-on-gap swallowed: `reconcile` step 4 discards `DrainResult.snapped` and refreshes `#lastDrainAt` (`predictor.ts:198, 216-221`), so a backgrounded tab never trips `SNAP_GAP_STEPS`. (4) Camera: centers on tile corner (missing +0.5 tile, `camera.ts:19-20`) and snaps to origin when own entity is unresolved (`main.ts:497-498`). (5) Pixi leak: `#drawMap` does `removeChildren()` without `destroy()` per warp (`render/world.ts:~74`). (6) `onUpdate` converts both rows fully just to compare two scalars (`connection.ts:109-126`).

- **12.5d-1:** THE remote interpolation delay and snapshot depth SHALL be consistent: either `INTERP_DELAY_STEPS = 1.0` with 2 snapshots, or keep 1.5 and deepen history to 3 — with a smoothness test that steady-cadence motion produces monotone positions (no hold/jump cycle).
- **12.5d-2:** WHEN a remote row's zone changes or its tile delta exceeds 1, THE store/upsert SHALL drop `prev` (snap, don't smear).
- **12.5d-3:** THE `snapped` signal SHALL be sticky across reconcile-drains until consumed by the frame loop (or gap tracking SHALL use last-FRAME-drain time).
- **12.5d-4:** THE follow camera SHALL center on the tile center and hold its last offset when the own entity is unresolved.
- **12.5d-5:** THE map renderer SHALL destroy replaced Graphics on zone switch; warp detection SHALL compare raw SDK row scalars without full conversion.
- Touches: `client/src/render/*`, `net/store.ts`, `prediction/predictor.ts`, `net/connection.ts`, `main.ts`. Same files as 12.5c — run 12.5c → 12.5d serially (or fold into one client slice).

### 12.5e — MEDIUM: battle lifecycle & rules residuals
Evidence: (1) Terminal `battle` rows are never deleted (only `battle_wild` is: `battle.rs:682`, `taming.rs:142`); table is public, client subscribes `SELECT * FROM battle` unfiltered (`connection.ts:86`), `start_battle` is un-rate-limited and permits self-battles (`battle.rs:78-83`) → client-driven unbounded public-table growth (ADR-0048 names the reaper as follow-up; still missing). (2) Self-battle XP farm: XP granted on any `SideAWins` regardless of provenance (`battle.rs:685+`), side-B takes no persistent damage, `heal_party` free/uncooldowned → riskless XP loop. (3) XP write-back uses `?` per-winner (`battle.rs:701-744`): one corrupt monster row rolls back the txn, battle stays Ongoing, the same failure re-fires every winning turn — the battle becomes unwinnable. (4) Owned-monster `known_skill_ids` order comes from `skill_row` table-scan order (`battle.rs:144-150`, `marshal.rs:303`) while wilds use canonical content order (`marshal.rs:130-161`); AI tie-break is first-seen (`ai.rs:48`) → enemy skill choice depends on storage scan order.

- **12.5e-1:** Terminal battles SHALL be GC'd — delete (or archive to a compact private row) at terminal write-back or via a scheduled reaper with a short TTL — while PRESERVING the client's battle-outcome frame (M8.7e renders outcomes from the row; keep the latest terminal battle per player, or delay deletion one reap cycle). Proof-of-teeth: an eval/test that a fled+won battle pair leaves ≤ the allowed residue.
- **12.5e-2 (DECIDED by Drew, 2026-07-03):** self-battle XP is ALLOWED as "practice XP" with a **0.1× multiplier** (90% penalty) applied to XP earned when the opponent is NOT `WILD_IDENTITY` (or future NPC provenance); wild/NPC battles keep full XP. Implement as follow-up sub-slice **12.5e-2-impl** (was deferred out of the M12.5e partial in flight at decision time). Rationale: preserves practice XP while removing the riskless-farm incentive. Implementation notes: multiplier applies at XP write-back by opponent provenance; keep the rule in `game-core` (SSOT); needs an ADR (game-design rule) + proof-of-teeth test that a self-battle win yields exactly floor/defined-rounding of 0.1× the wild-equivalent XP.
- **12.5e-3:** THE XP write-back loop SHALL log-and-continue per-monster on parse failure (matching `movement_tick`'s per-character philosophy) so a single corrupt row cannot make a battle unwinnable.
- **12.5e-4:** THE owned-monster path SHALL build `known_skill_ids` in canonical content order (mirror `wild_battle_monster`), removing the storage-scan-order dependence.
- Touches: `server-module/src/battle.rs`, `marshal.rs`, `schema.rs` (if reaper table/TTL), `evals/`. Serialize against M12 server slices.

### 12.5f — MEDIUM: gate & sim-harness teeth (extends 10.5d)
Evidence: (1) sim-harness runs on the in-code `zone_0()` stub with NO warp handling (`sim-harness/src/lib.rs:238`, `world.rs:80-98`) and the "drift gate" `map_for_zone_0_matches_zone_0_art` compares `ZONE_0_ROWS` to itself (`game-core/src/world.rs:461-494`) — authored-content drift and the entire M11 warp runtime have zero headless coverage. (2) Schema-snapshot eval records only column type NAMES (`"state": "BattleState"`), so a field added inside `BattleState`/`BattleMonster`/`MoveInput`/`EncounterEntryRow` — exactly the non-additive change ADR-0006 exists to catch — passes the gate. (3) `run.mjs` exits 0 when it finds zero evals (`run.mjs:10-17`) — fail-open on broken cwd/checkout (37 evals exist; empty can only be breakage). (4) `dom-shell-coverage-exclusion` matches shell paths against the whole config text incl. comments (`:31-33`). (5) Tautological `#[should_panic]`: `unknown_skill_id_panics` ends with an unconditional `panic!` (`resolve.rs:~1439-1458`) — passes regardless of behavior. (6) Recruit e2e: 7 `test.fixme` blocks whose stated revival condition ("until M9c lands") has EXPIRED; real blocker is the dev_reducers publish (`--bin-path`), and `main.ts:293` still omits `baitItems` so bait UI is dead in production; no revival guard covers `test.fixme` (spec-gap-revival covers only the Rust `#[ignore]`).

- **12.5f-1:** THE sim-harness SHALL build its maps from `load_zone_maps()`/`map_for` (real content), gain warp resolution in `tick_zone` mirroring `movement_tick`, and a convergence scenario crossing a warp tile; a REAL drift test SHALL compare `zone_0()` (if kept) against the authored RON.
- **12.5f-2:** THE schema-snapshot eval SHALL snapshot the field layout of `SpacetimeType` structs/enums reachable from table columns (re-baseline once), with a doctored-fixture tooth.
- **12.5f-3:** `run.mjs` SHALL exit non-zero when zero evals are found; `dom-shell-coverage-exclusion` SHALL strip comments (or scope matching to the `exclude:` array).
- **12.5f-4:** THE `unknown_skill_id_panics` test SHALL use `#[should_panic(expected = "…")]` without the trailing `panic!` (or catch_unwind + assert).
- **12.5f-5:** `spec-gap-revival.eval.mjs` SHALL also guard `test.fixme` anchors in `client/e2e/` (condition-expiry = RED), and the recruit-e2e revival + bait-UI wiring (`buildBattleViewModel` 4th arg + dev_reducers `--bin-path` publish in CI) SHALL be scheduled as a named slice (here or the runner's next infra slice — but named, not prose).
