## 1. Problem / intent

The seventh review confirms the spine remains sound (authority, RLS split-tables, determinism gate, additive schema, dual-write discipline, eval teeth all re-verified clean). The residual risk now clusters in four places: (1) the **gates that guard the gates** — the ci `just eval`/`just test` steps and the nightly mutation/coverage jobs can be deleted or neutered with zero mechanical bite, which silently darkens all 46 evals; (2) the **client feedback edge** — server reducer rejections are invisible to both the prediction layer (a verified silent 1-tile desync with `diverged=false`) and the user (dead buttons); (3) the **content lifecycle for the newest domains** — NPC edits never propagate (insert-only seeding), removed zones tick error logs forever, and hot paths re-parse the full RON registry per call; (4) **ledger drift** — PLAN/spec statuses under-report what shipped, and the harness↔project ADR numbering collision can resolve a bare "ADR-0055" to the wrong document.

Severity counts (all verified): 0 Critical, 1 High, 14 Medium, ~22 Low, 3 Info.

- **13.5a-3:** THE coverage threshold SHALL be re-measured post-exclusion and re-set a few points below actual (ADR-0050 ratchet policy re-applied); record the measurement in the ADR.
- **13.5a-4:** THE coverage-exclusion eval SHALL add `findUnsanctionedExclusions` (exact-set comparison vs the DOM_SHELLS allowlist + vitest defaults), with a bad fixture.
- **13.5a-5:** A structural check SHALL assert every `justfile` `ci:` dependency appears as a ci.yml step (minus documented substitutions, e.g. security→gitleaks/semgrep/cargo-audit).
- **13.5a-6 (DECIDED by Drew, 2026-07-04 — D-13.5-2):** THE nightly workflow SHALL gain a gating `cargo mutants -p server-module` job (sharded if runtime demands; NO continue-on-error), with survivor threshold set from an initial baseline run per the ADR-0050 ratchet policy; the new job SHALL be covered by the 13.5a-2 wiring/neuter guards from its first commit. ### 13.5b — MEDIUM (High player-visible value): reducer-rejection feedback — silent phantom-intent desync + dead-button UX + reconnect
Evidence (verified with repro): `predictor.ts:191` prunes `#pending` only by `p.seq > ackedSeq`; a server "queue full" reject (movement.rs:121-127) rolls back the seq ack (guards.rs:74-81 txn semantics); zero `conn.reducers.onX` callbacks exist anywhere in client/src, so the phantom op is replayed onto the auth queue at every reconcile → persistent 1-tile offset with `diverged=false` (reconcile compares before/after including the phantom) whenever the rejected send is the last input of a burst. Also: non-movement reducer Errs (heal cooldown, evolve ineligible, slot occupied) never reach the UI; `main.ts:607` sends healParty with `locationId ?? 0` when no locations are loaded (guaranteed invisible Err); `connection.ts:80` discards the SDK subscription-error payload; `connection.ts:106-109` still carries the M5 open question — onDisconnect only `store.reset()`, no confirmed auto-reconnect and no rebuild/backoff path (a drop leaves a blank world with live local prediction).

- **13.5b-1:** THE client SHALL register a reducer-status callback for `enqueue_move`; WHEN the reducer fails, THE pending op with that seq SHALL be dropped and a reconcile forced. Gating test per the repro sketch: enqueue+record seq N, reconcile repeatedly with ack N-1 → predicted position returns to authority (today it stays 1 tile off forever).
- **13.5b-2:** THE client SHALL register result callbacks for the non-movement reducers and route rejections through the existing `opts.onError` surface (toast/status line); the doomed healParty send SHALL be skipped when `healLocations()` is empty; `connection.ts:80` SHALL forward the SDK error payload.

### 13.5d — MEDIUM: content parse caching on hot paths (determinism-safe, shell-side)
