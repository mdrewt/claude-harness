## 1. Problem / intent

The eighth review confirms the spine remains sound (authority, determinism seams, additive schema discipline, economy/inventory arithmetic, gate wiring incl. the 13.5a gate-of-gates guards all re-verified clean). The residual risk clusters in the M14 delta itself: **M14 is complete on the happy path only.** (1) The **swap/recruit reducer paths bypass the entire post-turn status/weather pipeline** — no DoT, no weather chip, no status/weather countdown — and `attempt_recruit` still advances `turn_number`, so status/weather clocks freeze during recruit spam and swaps grant tick-free enemy turns (supersedes and closes the narrower named residuals R1/R3). (2) The **passive-ability system (M14c) is structurally inert in production** — `SpeciesRow` has no ability column, marshal hardcodes `ability: None`, and no reducer ever constructs an `AbilityStore` — while ARCHITECTURE.md declares abilities "fully built and merged". (3) The **client never surfaces two shipped M14 systems**: `use_battle_item` has no UI path at all (the M14e feature is player-unreachable), and weather is discarded in `rowConvert.ts` before the store (Sandstorm/Hail chip damage renders as unexplained HP loss). (4) A verified **status-misdirection bug**: Phase 4.5 writes `StatusApplied` to the post-switch active slot, so a weather/DoT faint between phases poisons the wrong (never-hit, just-switched-in) monster. (5) **Ledger drift**: the harness↔project ADR 0055–0057 collision note is factually wrong for 2 of 3 numbers, CHANGELOG is missing all of M14, and several stale claims/comments mislead (battle.rs:488 "load_skills() is cached" — it is not).

Severity counts (all verified @ `3eeb484`): 0 Critical, 5 High, 9 Medium, ~12 Low.

## 2. Slices, acceptance criteria (EARS) + located evidence (@ `3eeb484`)

### 14.5a — HIGH: swap/recruit battle paths run the full status/weather pipeline (closes R1, R3, RT-W14-DESYNC-01)
Evidence (verified, repro sketches on file): `resolve_player_swap` (resolve.rs:569) and `resolve_recruit_failure` (resolve.rs:547) bottom out in `resolve_enemy_turn`→`resolve_one_attack` only — `apply_post_turn_effects`, `apply_weather_damage`, `tick_status`, `tick_weather` are unreachable from either (only `resolve_full_turn`, called solely by `submit_attack`, runs Phases 3/3.5/4/5). `resolve_recruit_failure` calls `advance_turn` (resolve.rs:561), so turn_number advances while every clock freezes. `swap_active` (battle.rs:558-619) and `attempt_recruit` (taming.rs:165) discard events (`let _events = …`), build no `BattleStatusStore`; `resolve_player_swap`'s signature structurally cannot persist StatusApplied. `attempt_recruit` additionally resolves retaliation via `skill_defs_from_rows` (marshal.rs:392-393 hardcodes `sets_weather: None, applies_status: None`) while battle.rs uses `load_skills()` — the RT-W14-DESYNC-01 red-team test pins this divergence.

- **14.5a-1:** THE game-core swap/recruit resolvers SHALL accept `&mut BattleStatusStore` and run the same post-turn pipeline as `resolve_full_turn` (enemy retaliation → StatusApplied write-back → DoT → weather chip → status tick → weather tick), with faint/auto-switch cascades handled identically; `swap_active` and `attempt_recruit` SHALL build/persist the store exactly as `submit_attack` does.
- **14.5a-2:** WHEN a failed recruit advances `turn_number`, THE status and weather clocks SHALL tick that turn (gating test: Sandstorm turns_remaining decrements and chip damage lands across `resolve_recruit_failure`; a poisoned monster takes DoT across `resolve_player_swap`).
- **14.5a-3:** THE recruit retaliation path SHALL use the same skill source as battle (`load_skills()` / the 14.5e cached accessor), retiring `skill_defs_from_rows`'s battle use or extending `skill_row` — RT-W14-DESYNC-01 flips from documenting the bug to pinning the fix.
- **14.5a-4 (DECISION D-14.5-1, default (b) if unresolved):** ADR-0092 §D3 says a status-blocked side's Swap is converted to Pass, but `swap_active` never consults `check_action_block` — Sleep/Paralysis/Freeze are always escapable by switching, and the D3 swap-conversion branch is dead in production. Either (a) enforce the block in `swap_active`, or (b) bless always-swappable and amend ADR-0092 D3 + add the pinning test. Gameplay judgement; (b) is the Pokémon-conventional behavior and cheaper.
- Touches: `game-core/src/combat/resolve.rs`, `server-module/src/battle.rs`, `server-module/src/taming.rs`, `server-module/src/marshal.rs`, tests. Disjoint from deflake-recruit-r2 (e2e-only).

### 14.5b — MEDIUM: Phase 4.5 status write targets the slot hit, not the post-switch active slot
Evidence (verified + repro): resolve.rs Phase order is 3 DoT (452-457) → 3.5 weather chip (460-464) → 4 tick (467-471) → 4.5 store write (474-486, reads `state.side_X.active` AT WRITE TIME) → 5. Both `apply_post_turn_effects` (status.rs:211-247) and `apply_weather_damage` (weather.rs:226-247) can faint + auto-switch between Phase 2's StatusApplied emission and Phase 4.5's write — the freshly switched-in bench monster receives the status earned by the fainted target. The safety comment at types.rs:213-224 only covers the Phase-2 same-attack case. No test combines same-turn StatusApplied with a DoT/weather faint cascade.

- **14.5b-1:** THE StatusApplied event SHALL carry the target slot index captured at emission; Phase 4.5 SHALL write to that slot only if it still holds a conscious monster (else drop the event); the types.rs safety comment SHALL be corrected.
- **14.5b-2 (proof-of-teeth):** a red-team test SHALL pin it: near-lethal status hit + Sandstorm chip faint in one turn → bench switch-in has NO status (RED today).
- Touches: `game-core/src/combat/{resolve,types,weather,status}.rs`, tests. Small, contained.

### 14.5c — HIGH: wire the passive-ability system end-to-end (or explicitly de-scope the claim)
Evidence (verified end-to-end): `SpeciesRow` (schema.rs:78-90) has no ability column; content write phase (content.rs:126-137) drops `sp.ability`; marshal (marshal.rs:276) hardcodes `ability: None`; zero non-test call sites for `AbilityStore`/`apply_entry_ability`/`apply_ability_modifiers`; `game-core/content/abilities/000-core.ron` defines 3 abilities but no species references one — and `validate_abilities` would happily accept a species that did, which would then silently do nothing. ADR-0095 deferred wiring "to m14e"; ADR-0096 didn't do it; ADR-0097's residuals table omits it; ARCHITECTURE.md:939-948,970-971 claims abilities are delivered.

- **14.5c-1:** THE `species_row` SHALL gain an additive `ability` column populated from content; marshal SHALL thread it; schema-snapshot baseline updated (additive).
- **14.5c-2:** THE battle reducers SHALL construct an `AbilityStore` from species content at battle start and on every switch-in (including 14.5a's swap paths), calling `apply_entry_ability` on entry and `apply_ability_modifiers` in `resolve_full_turn` per the wiring described in ability.rs:9-17 doc comments and ADR-0094.
- **14.5c-3:** AT LEAST one shipped species SHALL exercise each ability kind (content change), pinned by an integration test (e.g. Flame Body species cannot be Burned; EntryHeal heals on switch-in).
- **14.5c-4:** IF de-scoped instead (DECISION D-14.5-2, default: wire it — the docs claim delivery), THE ADR-0097 residuals table, ARCHITECTURE.md M14c/Phase-B lines, and a content-validation guard (reject species with `ability: Some(_)` until wired) SHALL land instead.
- Touches: `server-module/src/{schema,content,marshal,battle}.rs`, `game-core/src/combat/resolve.rs`, content RON, `evals/baselines/`, bindings regen, tests. STRUCTURAL (schema) → serial slice, no fan-out.

### 14.5d — HIGH: client battle UX completeness — cure items reachable, weather visible
Evidence (verified): no `onUseItem` callback, no item UI, zero client references to `useBattleItem` — the shipped M14e feature is player-unreachable (main.ts:838-851, battleView.ts:9-21). Weather: generated bindings carry `BattleState.weather` (module_bindings/types.ts:93) but `SdkBattleRow.state` (rowConvert.ts:173-186) omits it, `battleRowToStore` never reads it, `StoreBattle` has no field, zero client references — chip damage is unexplained HP loss. Supporting gaps: `statusBadge` has zero tests and a silent-empty default arm; `BattleViewModel.outcome` is bare `string`, so a new server `BattleOutcome` variant compiles silently.

- **14.5d-1:** THE battle overlay SHALL offer a Use-Item action listing owned items with `cure_status` (classify-by-data, mirroring the bait-selector pattern), wired to the `use_battle_item` reducer; e2e or component test covers cure of a poisoned active monster.
- **14.5d-2:** THE weather state SHALL be threaded `rowConvert → StoreBattle → BattleViewModel` and rendered as a field banner (kind + turns remaining), parallel to the status badge.
- **14.5d-3:** `statusBadge` SHALL gain a unit test asserting every Rust `StatusEffect` variant maps to a non-empty badge (parity guard, js-path-parity-style); `outcome` SHALL become the literal union of `BattleOutcome` variants so new variants are compile errors.
- **14.5d-4:** THE battle-overlay refresh guard (13.5e-1 VM compare) SHALL include the new status/weather fields so the banner doesn't churn on unrelated batches.
- Touches: `client/src/**` only. Disjoint from all server slices; serialize vs deflake-recruit-r2 only if it touches `client/e2e` recruit flows.

- **14.5e-2:** Parity/determinism evals stay green (cache must be observationally transparent).
- Touches: `server-module/src/{content_cache,battle}.rs`. Tiny, disjoint.

### 14.5f — MEDIUM: gates — prove additive-schema at the real codec; widen the convergence net
