## 1. Problem / intent

The ninth review confirms the M15 trading spine is well-built where prior reviews taught the patterns (escrow guards wired at all 12 spend sites, atomic confirm-time ownership re-read with rollback, clean pure/shell split for the swap engine, additive schema with updated baselines, server-authoritative client render with no parallel trade state). The residual risk clusters at the **seams M15 did not close**: (1) the battle↔trade interlock is one-directional — `start_battle` rejects escrowed monsters but **nothing stops trading away a monster that is in an ongoing battle**, after which every battle-terminal path aborts at `write_back_party_hp`'s owner-mismatch check and rolls back, leaving a permanent Ongoing zombie battle that is never GC'd and permanently blocks the original player from battling (verified repro on file); the one test anchoring that abort invariant (`m7b_2_owner_change_mid_battle_spec_gap`) was revived as a **zero-assertion comment body** that the revival gate cannot see through. (2) `confirm_trade` debits the sender in full but **credits through silently-clamping `grant_item`/`grant_currency`** — at the receiver's stack/balance cap, traded value is destroyed with no error. (3) The **trade reducer bodies are executed by no test at any layer** — coverage is pure-function units + token-presence source scans (inversion-blind for the respond/confirm status checks), the spec's own "e2e propose→respond→confirm" DoD line is unmet, and `attempt_recruit` is the one escrow site absent from the escrow-guards eval. (4) Client: the trade overlay's mutual exclusivity is **asymmetric** (KeyQ/KeyH/KeyG stack their overlays on top of an open trade, contradicting ADR-0107 D5), and `propose_trade` has no client entry point (recorded deferral). (5) Ledger drift: ADR README next-free points at an existing file (in-flight PR #171 claims the fix — verify), ARCHITECTURE.md calls shipped M15c "PARKED" and its canonical module table omits `economy.rs`/`trading.rs`/`content_cache.rs`, CHANGELOG ends at ADR-0103, and the harness PLAN/spec checkboxes lag by two milestones.

Severity counts (all verified @ `3424c5c`): 0 Critical, 1 High, 7 Medium, 12 Low.

## 2. Slices, acceptance criteria (EARS) + located evidence (@ `3424c5c`)

### 16.5a — HIGH: close the battle↔trade interlock both ways + give the owner-change abort real teeth
Evidence (verified, full repro on file): `trading.rs` contains zero battle checks (`grep -n battle server-module/src/trading.rs` → one doc comment); `propose_trade` (97-234) and `confirm_trade` (278-392) validate ownership only, and confirm unconditionally reassigns `owner_identity` + clears `party_slot` (357-366). `start_battle` guards the other direction (battle.rs:143-152 `reject_if_monster_in_trade`). `write_back_party_hp` aborts on owner mismatch (battle.rs:940-948) and ALL terminal paths (`submit_attack`:657-659, recruit arm:783, `flee`:806-809) route through it BEFORE the row update, so the Err rolls back the terminal outcome → battle stays Ongoing forever (GC is terminal-only, battle.rs:987-992; the only scheduled table is movement_tick_schedule) → `already_in_battle` (battle.rs:109-116) bricks the player permanently. The in-code claim that escrow guards make this "unreachable" (battle.rs:940 comment) is false. Meanwhile `m7b_2_owner_change_mid_battle_spec_gap` (game-core/src/combat/m7b_redteam_tests.rs:157-178) is un-ignored with a body of pure comments — zero assertions — and `evals/spec-gap-revival.eval.mjs` (specGapStatus) checks only un-ignored + fn-name-present, so the gate is green while the abort branch has NO covering executable test and NO source-scan eval anywhere.

- **16.5a-1:** WHEN `propose_trade` is called with any offered monster (either side) that is in an `Ongoing` battle, THE reducer SHALL reject (reject-not-clamp), using the same owner-scoped `reject_if_in_battle` pattern as `evolve`/`fuse` (evolution.rs:66-72); `confirm_trade` SHALL re-assert the check before `build_swap_plan` (defense-in-depth, mirroring its ownership re-read).
- **16.5a-2:** THE check SHALL cover the PvP battle rows landed by m16a (a monster can be on the `opponent_identity` side of a shared PvP battle row — the M15-era owner-scoped iterator does not see that side; extend the iterator or the guard accordingly). Rebase this slice onto merged m16a before writing tests.
- **16.5a-3 (proof-of-teeth):** a red-team test SHALL pin the closed hole: start battle with monster M → propose_trade(M) MUST Err (RED today); plus the existing-style teeth mutation check that deleting the new guard flips a gate RED.
- **16.5a-4:** `m7b_2_owner_change_mid_battle_spec_gap` SHALL gain a real body: construct a battle whose party monster's `owner_identity` differs from `battle.player_identity`, call `write_back_party_hp`, assert `Err` AND the other player's monster row untouched (verifier's minimal test sketch is on file; if a ReducerContext harness is impossible in-crate, add a trade-eval-style source-scan asserting the abort branch instead — but prefer the real test).
- **16.5a-5:** `evals/spec-gap-revival.eval.mjs` SHALL additionally reject a revived anchor whose stripped body contains no `assert` (closes the vacuous-revival class, with a bad-fixture proof-of-teeth).
- Touches: `server-module/src/{trading,battle}.rs`, `game-core/src/combat/m7b_redteam_tests.rs`, `evals/spec-gap-revival.eval.mjs`, tests. Overlaps M16 battle surface → serialize after m16a merge.

### 16.5b — MEDIUM: conservation at receiver caps — reject, don't destroy
Evidence (verified): `confirm_trade` (trading.rs:371-388) debits via fallible `consume_one`/`spend_currency` but credits via infallible `grant_item` (inventory.rs:44-59, silent no-op/clamp at `MAX_ITEM_STACK` 9999) and `grant_currency` (economy.rs:28-44 → currency.rs:17 `saturating_add(...).min(MAX_BALANCE)`). No headroom check exists in propose or confirm; `build_swap_plan` checks only ownership. No test touches the cap boundary (grep of trading_tests.rs for MAX_ITEM_STACK/MAX_BALANCE/9999 → nothing). Trading 50 potions to a player holding 9,980 destroys 31 with no error.

- **16.5b-1:** WHEN executing a confirmed trade, IF any item credit would exceed the receiver's `MAX_ITEM_STACK` headroom or any currency credit would exceed `MAX_BALANCE` headroom, THEN `confirm_trade` SHALL return `Err` and roll back (reject-not-clamp; no partial swap). Optionally pre-check at `propose_trade` for better UX (non-authoritative; the confirm-time check is the invariant).
- **16.5b-2 (proof-of-teeth):** boundary tests in `trading_tests.rs`/reducer-level: receiver at 9,980 of an item receiving 50 → Err, both inventories unchanged; currency analog at MAX_BALANCE. Extend `evals/trade-conservation.eval.mjs` to require the headroom-check call sites.
- Touches: `server-module/src/trading.rs`, `game-core/src/trading/rules.rs` (if the check goes in `build_swap_plan` — preferred, it's pure), tests, one eval. Small, contained.

### 16.5d — MEDIUM: trade runtime coverage — execute the reducer bodies somewhere (DECISION D-16.5-1)
