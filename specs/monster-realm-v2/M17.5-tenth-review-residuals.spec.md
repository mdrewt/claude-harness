## 1. Problem / intent

The tenth review confirms the M15–M17 delta is architecturally disciplined: schema changes are strictly additive with matching baselines, new Elo/forfeit/TTL rules sit in game-core behind thin reducers, determinism boundaries hold, the m17b client is a pure subscription view, and the eval generation ships bidirectional proof-of-teeth fixtures (ADR-0118's mutant-cap 180→309 re-baseline was verified as an honest re-measurement, not a gate-weakening). The residual risk clusters at the **seams between hardening waves** — invariants proven for one code path but not propagated to its siblings:

1. **The "one ongoing battle per player" invariant is enforced against only the `player_identity` role.** `start_battle`, `begin_encounter`, `heal_party`, `evolve`, and `fuse` each scan `battle().player_identity().filter(me)` only — while `pvp::is_in_ongoing_battle` (the M16 guard) correctly checks BOTH `player_identity` and `opponent_identity`. A PvP battle indexes side B under `opponent_identity`, so the accepting player can open a **second concurrent battle** (production-reachable via a grass encounter → `begin_encounter`, or a self-practice `start_battle`) with the same party monsters — enabling a **PvP-damage-laundering exploit** (snapshot HP in the wild battle, lose the PvP battle, flee the wild battle to restore pre-PvP HP) plus mid-PvP heal and evolve/fuse of a battling monster. ADR-0119 documented `m17-fix-sideb-guards` as an unscheduled candidate slice covering `start_battle`/`begin_encounter`/`movement_tick`/`heal_party`, but it was never scheduled, and `evolve`/`fuse` are not in that list. This slice schedules and widens it.

2. **Value-conserving credit is not uniform.** `confirm_trade` debits the sender in full but, for a same-item bilateral swap where the receiver is at/near `MAX_ITEM_STACK`, applies the credit BEFORE the offsetting debit — and `grant_item` clamps monotonically (drops the whole grant at cap), so the net-of-send headroom check passes while items are silently destroyed. The same silent-clamp class exists un-guarded in `buy`/`sell` (grant at `MAX_ITEM_STACK`/`MAX_BALANCE` with no headroom reject). This is exactly the "reject, don't destroy" invariant ADR-0113 established for the trade receiver-cap path, not propagated to the same-item-swap ordering or the shop paths.

3. **PvP — the flagship multiplayer feature — has no runtime coverage above unit math.** Challenge/accept/turn/forfeit orchestration, the deadline reaper, and the both-directions battle↔trade interlock (m16.5a) are guarded almost entirely by source-grep static checks and in-crate unit tests; there is no two-context e2e and the sim-harness explicitly excludes forfeit/deadline. Separately, the client's SDK-boundary enum fields are typed `{ tag: string }` (with a cast at `rowConvert.ts:525`), so a server-added enum variant regenerates bindings cleanly and passes tsc, drift, and unit gates while the client silently mishandles the unknown tag.

4. **Two user-facing correctness gaps and a stale accepted-risk pointer.** `profile.name` is seeded once and never refreshed, so the public leaderboard permanently shows a player's first-ever name after any rename (the `player` row is recreated each `join_game`). `battle_challenge` rows have no TTL (asymmetric with trade offers post-ADR-0117) and no challenge cooldown, allowing a pending challenge to a disconnected target to persist indefinitely and challenge-spam to lock a victim's slot. And the world-readable table doc-comments/RLS deferral pointers still say "→ M16" — a milestone that shipped PvP, not transport RLS — leaving the exposure open-ended while the docs imply it was time-boxed.

5. **Ledger drift.** CHANGELOG.md ends at #192 (8 merges behind, missing both M17 feature milestones); ARCHITECTURE.md's canonical module map (used for `touches:` disjointness calls) omits `use_battle_item` and `is_ranked_pvp`; PLAN.md §Status and M17 §5 slice merge-states lag the merge stream; the ADR README index mislabels ADR-0107 as a "T-key" overlay (it is KeyU); and two game-core trading tests carry doc-comments claiming live vulnerabilities that are already fixed.

Severity counts (verified @ `9a74e2a`): 0 Critical, 2 High, 8 Medium, ~14 Low. This milestone bundles the verified, no-new-game-design hardening (a–g). Four items are surfaced as **decisions** for Drew rather than flat fixes (see §3).

## 2. Slices, acceptance criteria (EARS) + located evidence (@ `9a74e2a`)

### 17.5a — HIGH: unify the "already in an ongoing battle" guard to both battle roles (close the side-B laundering exploit)
Evidence (verified by direct read): `start_battle` (battle.rs:110-116), `begin_encounter` (battle.rs:332-339), and `heal_party` (raising.rs:283-292) each check `battle().player_identity().filter(me).any(|b| ...Ongoing)` only; `evolve` (evolution.rs:66-69) and `fuse` (evolution.rs:216-222) pass only the `player_identity()` iterator to `reject_if_in_battle`. `pvp::is_in_ongoing_battle` (pvp.rs:96-110) already checks both indexes (with the `!= WILD_IDENTITY` refinement). PvP battles index side B under `opponent_identity`, so a side-B player passes every PvE-path check. `begin_encounter` is reached in production from `movement_tick` (movement.rs:276); `start_battle` is the player-callable practice path. (`start_wild_battle` shares the gap but is `#[cfg(feature = "dev_reducers")]`, so it is not a prod vector.) The laundering path: side B walks into grass mid-PvP → concurrent wild battle snapshots current HP → loses PvP → `flee` restores pre-PvP HP via `write_back_battle_results`.

- **17.5a-1:** THE ongoing-battle precondition in `start_battle`, `begin_encounter`, and `heal_party` SHALL reject when the caller is in an `Ongoing` battle in EITHER the `player_identity` or the `opponent_identity` role, reusing a single shared helper (hoist/reuse the `pvp::is_in_ongoing_battle` predicate into `guards.rs` so all callers — PvE and PvP — share one definition; keep the `!= WILD_IDENTITY` refinement so a wild/practice battle's own side-A owner is still caught but the `WILD_IDENTITY` opponent sentinel is not spuriously matched).
- **17.5a-2:** THE `reject_if_in_battle` calls in `evolve` and `fuse` SHALL cover both battle roles for the monster's owner (chain the `opponent_identity` iterator, mirroring the m16.5a trade-interlock fix in `trading.rs`).
- **17.5a-3 (proof-of-teeth):** a red-team test SHALL construct a PvP battle, then assert that side B calling `start_battle` / `begin_encounter` (via the movement or a test seam) / `heal_party` / `evolve` / `fuse` is rejected (RED against current code); plus a mutation check that deleting the `opponent_identity` branch flips a gate RED. Include one test asserting the laundering path is closed (side B cannot hold two `Ongoing` battle rows).
- **17.5a-4:** confirm `movement_tick`'s encounter trigger inherits the fix through `begin_encounter` (no separate guard needed) and note it in the ADR.
- Touches: `server-module/src/{battle,raising,evolution,guards}.rs` (+ `*_tests.rs`), `evals/battle-reducer-security.eval.mjs` (optional needle). Structural on the battle path → **SERIAL**. This subsumes and widens the ADR-0119 `m17-fix-sideb-guards` residual.

### 17.5b — HIGH→MEDIUM: trade same-item near-cap swap silently destroys items (conservation)
Evidence (verified): `confirm_trade` applies `plan.item_transfers` in order, each doing `consume_one(from)` then `grant_item(to)` (trading.rs:618-630); `grant_item` (inventory.rs:44-47) is monotone-clamping — `if row.count < MAX_ITEM_STACK { ... .min(MAX_ITEM_STACK) }`, so a grant to a stack already at 9999 is dropped entirely. The net headroom check (trading.rs:531/557) nets `raw_count.saturating_sub(sending_qty)`, which is exact only if all debits precede all credits — but for a same-item bilateral swap the counterparty's credit (transfer 1, initiator-items-first ordering) lands before its debit (transfer 2). Repro: counterparty at 9999 of item X; offer gives 15 X and takes 20 X → headroom sees 9999−20+15=9994 ≤ 9999 → passes; apply drops the 15 credit at cap then debits 20 → final 9979, 15 destroyed silently. `MAX_ITEM_STACK = 9999` (game-core/src/trading/rules.rs:15). The initiator side is safe only by transfer ordering. The currency headroom uses raw (un-netted) balances (trading.rs:559-560), a latent false-reject that must be co-fixed if the apply order changes.

- **17.5b-1:** WHEN `confirm_trade` executes, all debits (both parties' `consume_one` and `spend_currency`) SHALL be applied before any credit, so the net-of-send headroom check is exact and no same-item swap can hit a transient over-cap. (Alternative/belt-and-braces per reject-not-clamp: route trade credits through a fallible `grant_item_exact`/`grant_currency_exact` that returns `Err` instead of clamping, rolling back the transaction; the debits-first reorder is preferred as it also fixes the false-reject.)
- **17.5b-2:** IF debits-first is adopted, THE currency headroom inputs SHALL be netted (`balance − outgoing`) symmetrically with the item stacks, so a cap-adjacent currency swap is neither falsely rejected nor able to destroy value.
- **17.5b-3 (proof-of-teeth):** regression tests: bilateral same-item swap with receiver at 9999 (and 9998) → assets conserved exactly, no Err on a legal net; a swap whose net genuinely exceeds cap → Err with both inventories unchanged.
- Touches: `server-module/src/trading.rs`, `game-core/src/trading/rules.rs` (if the check/plan moves), `trading_tests.rs`, one eval. Contained.

### 17.5c — MEDIUM: shop `buy`/`sell` — reject, don't destroy, at receiver caps
Evidence (verified): `buy` does `spend_currency(me, total)?` then infallible `grant_item(me, item_id, qty)` (economy.rs:143-147) — at `MAX_ITEM_STACK` the player pays full price and receives nothing; `sell` does `consume_one × qty` then `grant_currency(me, total)` (economy.rs:224-227) — at `MAX_BALANCE` items are consumed for clamped proceeds. No headroom reject on either path; `currency-integrity`/economy evals don't catch it.
- **17.5c-1:** `buy` SHALL reject (before spending) when `current_item_count + qty > MAX_ITEM_STACK`; `sell` SHALL reject (before consuming) when `balance + total > MAX_BALANCE`. Reuse/extend `game_core::check_headroom`.
- **17.5c-2 (proof-of-teeth):** boundary tests for both paths; extend the economy eval to require the headroom call sites.
- Touches: `server-module/src/economy.rs` (+tests), `game-core/src/trading/rules.rs` (shared headroom), one eval. Small.

### 17.5d — MEDIUM: leaderboard `profile.name` drift
Evidence (verified): `get_or_init_profile` (ranking.rs:27-46) sets `name` only in the insert arm; `apply_pvp_rating`'s `update(Profile { rating, ..winner })` spreads keep the old name; `join_game` (movement.rs:44) recreates the `player` row with a fresh caller-supplied name each session; `player` rows are deleted on disconnect (lib.rs:197). No path updates `profile.name`, so the public leaderboard permanently shows the first-ever rated name.
- **17.5d-1:** WHEN `apply_pvp_rating` writes an updated profile (or in `get_or_init_profile`'s `Some` arm), THE reducer SHALL refresh `name` from the live `player` row when present, before the update.
- **17.5d-2:** a test SHALL assert a rename-then-rated-game surfaces the new name on the profile. (Relates to the parked `set_profile_name`/RL-14 slice — see §3 Decision 3; this slice only keeps the passive mirror honest, it does not add a name-edit reducer.)
- Touches: `server-module/src/ranking.rs` (+tests). Small; SERIAL with 17.5a only if both touch ranking/battle.

### 17.5e — MEDIUM: `battle_challenge` liveness — TTL reaper + anti-spam cooldown
Evidence (verified): trade offers gained a per-row one-shot TTL reaper in m16.5f (trading.rs, ADR-0117); `battle_challenge` has only `cancel_challenges_on_disconnect` for OUTGOING rows (ADR-0109 D9 deliberately keeps incoming ones), no TTL, and no per-identity challenge cooldown — a Pending challenge to an offline target persists indefinitely (challenger locked out of new challenges), and repeated re-challenge after each decline spams the victim's UI and blocks their own outgoing-challenge slot (pvp.rs guard 5b).
- **17.5e-1:** `battle_challenge` rows SHALL expire via a per-row one-shot TTL reaper cloned from the `trade_offer_reaper` pattern (schedule-table + btree-indexed disarm on delete; TTL constant in game-core). Choose a challenge TTL (decision D-17.5-1, default e.g. 2 min) and record it.
- **17.5e-2:** THE `challenge_pvp` reducer SHOULD apply a short per-identity re-challenge cooldown after a decline (or a decline-and-mute primitive deferred to M19 — record the choice).
- **17.5e-3 (proof-of-teeth):** reaper-fires + disarm-on-accept tests; a source-scan eval pinning the reaper wiring (mirror the trade reaper eval).
- Touches: `server-module/src/{pvp,schema}.rs` (+tests), `game-core` const, one eval. Schema-additive → coordinate with any in-flight schema slice.

## 3. Decisions for Drew (present as choices, not flat recommendations)

> **Resolved 2026-07-17** per Drew's playtest-replan mandate — see `playtest-replan-2026-07.md` §3:
> A=adopt BattleKind in 17.5 · B=re-book RLS to M22 + fresh accepted-risk note · C=schedule → M-playtest-c
> · D=schedule → M-playtest-c · E=schedule → M-playtest-a. Veto/amend by editing that doc.

- **D-17.5-A — `is_ranked_pvp` classification by sentinel vs an explicit `BattleKind` column.** Today "ranked" is negative-space (`!= self && != WILD_IDENTITY`), so any future battle source with its own opponent identity (NPC trainers, tournaments, friendlies) silently rates as ranked. Adding an additive `kind: BattleKind` enum column now is cheap under the additive-schema regime and makes the classification exhaustive/compile-checked; deferring keeps the surface smaller but the fix gets more expensive per new battle type. **Decide:** add the column in this milestone (recommend if any non-wild non-ranked battle is on the M18+ roadmap) or accept the sentinel with a guarding eval.
- **D-17.5-B — transport RLS: re-book under a live milestone vs indefinite accepted risk.** All runtime tables use unfiltered `SELECT *` subscriptions (inventory, battle-with-both-teams-in-a-blob, trade_offer, profile, …). The deferral has pointed at "M16" since before M16 shipped PvP; the `my_conversation` owner-scoped view pattern proves the mechanism now exists. **Decide:** (a) re-book owner-scoped views + de-`public` as a real future milestone (M20 observability/M22 privacy are natural homes), or (b) record a fresh ADR explicitly re-accepting world-readability with the view mechanism acknowledged. 17.5g-3 only fixes the stale pointer; the underlying scope is this decision.
- **D-17.5-C — `set_profile_name` / RL-14 name-edit slice.** m17b's UI and its test allowlists reference a `set_profile_name` write that does not exist server-side; the follow-up (`m17b-2`) is named only in handoffs, absent from any spec/PLAN/§5 row. **Decide:** schedule it (server reducer + `validate_name` at the profile-write layer + RL-7 tooth amendment + battleModel/View delta), or record an explicit park. 17.5d keeps the passive name mirror honest regardless.
- **D-17.5-D — `propose_trade` client entry point.** M15's headline feature is human-unreachable on the propose side (only via the `__mrTrade` test hook); M16.5 recorded "do not silently drop it." **Decide:** schedule the thin propose-UI slice (tradeView form → `reducers.proposeTrade` + e2e) or explicitly park to a named milestone. Also **D-17.5-E:** gate `__game`/`__mrTrade`/`__mrPvp` behind `import.meta.env.DEV` (the in-code comment asks for it; ADR-0115 accepted them ungated — reconcile the contradiction one way).
