## 1. Problem / intent

A read-only, multi-lens review of the delivered spine (M0–M8a + M-infra-a) found the architecture
**sound** — the functional-core/imperative-shell spine, rule-SSOT + wasm parity, determinism gate,
privacy split-table, and reconcile core all hold — but surfaced a hardening backlog that should be
cleared **before the game grows further** (M9+ builds directly on this code):

1. **One real authorization/privacy gap** at the battle entry point: `start_battle` trusts a
   client-supplied `opponent_identity` + monster-id list, so a player can conscript *another player's*
   monsters into a `public` battle row (info disclosure + griefing + XP fabrication).
2. **Latent correctness/contract gaps** in the rule core (divide-by-zero on `defense==0`, `turn_number`
   overflow, silent `as u16` truncation) — not reachable today but cheap to make impossible by construction.
3. **Toothless gates** — two "proof-of-teeth" evals don't actually bite the property they guard
   (`battle-reducer-security` would pass the §1.1 bug green; the monster/`monster_pub` dual-write has no
   mechanical test).
4. **Claimed-but-unwired mechanical enforcement** — the formatter gate (`cargo fmt`/`biome`) isn't in
   `just lint`; mutation + coverage are in the "Done" definition but not wired anywhere.
5. **Documentation/ADR drift** — the XP formula, the damage `u64`/`u32` story (ADR-0041 asserts an overflow
   "can't happen" that *can*), a false "client predicts battles" doc, stale CHANGELOG, dangling ADR
   cross-refs.

Success = every P0–P2 finding resolved with a falsifiable gate where one applies; the doc set matches the
code; no game-design surface added; `just ci` green-and-meaningful. This milestone is **pure hardening** —
it changes no player-facing behavior except closing the `start_battle` abuse path.

## 3. Acceptance criteria (EARS)

**Battle security & integrity (server-module; ADR-0042 amend + proposed ADR-0048)**
- WHEN `start_battle` is called THE SYSTEM SHALL accept opponents only from a **server-authored/NPC
  provenance** (or `opponent_identity == ctx.sender` for self/sandbox) and SHALL reject with `Err` any
  `opponent_identity` resolving to another live player's monsters — it SHALL NOT trust a client-supplied
  opponent roster (reject-not-clamp).
- WHEN `start_battle` validates the caller's party THE SYSTEM SHALL reject (`Err`) if
  `party_monster_ids.len() > MAX_PARTY_SIZE` or if any listed monster is not party-slotted, in addition to
  the existing ownership + duplicate-id checks.
- WHEN `write_back_battle_results` runs THE SYSTEM SHALL assert `side_a.team.len() ==
  party_monster_ids.len()` and index via checked access (`.get(i)`), returning `Err` rather than
  panic-indexing on a length mismatch.
- AFTER any battle ends THE SYSTEM SHALL leave every side-B (opponent) `monster` and `monster_pub` row
  **byte-for-byte unchanged** (the side-B no-write is intentional PvE semantics; recorded in ADR-0042).
- PROOF-OF-TEETH: `evals/battle-reducer-security.eval.mjs` SHALL assert *authorization behavior*, not
  substring presence — it SHALL fail a fixture where `start_battle` accepts a non-consenting player's
  monsters, and fail a fixture where an outcome guard merely *reads* `.outcome` without comparing it
  (require an adjacency to `==`/`!=`/`BattleOutcome::Ongoing`).

**Rule-core contracts (game-core; debate-resolved, proposed ADR-0049)**
- THE `calc_damage` contract SHALL document `defender.stats.defense >= 1` as a precondition and SHALL
  `debug_assert!(defense > 0)`; the divisor SHALL NOT be silently clamped. WHEN a `BattleMonster` is built
  at the trust boundary (`battle_monster_from_row`) THE SYSTEM SHALL reject/abort a row whose derived
  defense is `0` (make the illegal state unrepresentable at the boundary, not inside the pure formula).
- WHEN `resolve_turn` would advance `turn_number` past `u16::MAX` THE SYSTEM SHALL terminate the turn at a
  defined terminal outcome rather than panic (debug) / wrap (release).
- WHEN `derive_stats` produces a value exceeding `u16` THE SYSTEM SHALL saturate explicitly
  (`u16::try_from(..).unwrap_or(u16::MAX)`) or assert the `base <= 255` precondition — never silently
  wrap via `as u16`.
- THE base-stat-total used by the XP rule SHALL be defined by a pure `game-core` function (the rule layer
  owns the BST definition); the server helper SHALL only marshal rows into it.
- THE "panic for a content-integrity invariant in the pure core" pattern (e.g. skill-not-found in
  `resolve_one_attack`) SHALL be recorded as a deliberate, gated decision (ADR/`doc-comment`) naming the
  `validate_content` referential-integrity check that guarantees it never fires in production.

**Gate teeth & test rigor (evals/tests/CI)**
- PROOF-OF-TEETH: an eval/test SHALL assert the **dual-write invariant** — for every monster-mutating
  reducer, the `monster_pub` projection equals `pub_from_monster(monster_from_instance(..))` of the
  written `monster` (field parity on safe fields); it SHALL fail a fixture that writes `monster` without
  mirroring `monster_pub`.
- WHEN a "spec-gap anchor" test documents an unclosed gap THE test SHALL be RED until the gap closes
  (`#[ignore = "spec gap: …"]` over a real failing assertion or a compile-gated marker) — no
  `assert!(true)` anchors SHALL remain.
- WHEN the `ci` GitHub job runs THE `bindings-drift` eval SHALL execute against an installed `spacetime`
  CLI (or fail loud when committed bindings exist but the CLI is absent) — drift SHALL NOT be catchable
  only in the slower `e2e` job.
- WHEN the `cache-freshness` sccache criterion is evaluated it SHALL be co-location-aware (check the
  `cache-on` recipe body) and SHALL pass legitimate non-sccache setups rather than failing on string
  absence.
- WHEN mutation/coverage are claimed THE definition SHALL match reality: `cargo mutants` (game-core, with a
  caught-mutant threshold) and a vitest line-coverage threshold SHALL run in a **scheduled/per-milestone**
  job, and AGENTS.md + the PR template SHALL state exactly what `just ci` enforces vs what the nightly job
  enforces.

**Build/CI/toolchain hygiene**
- WHEN `just lint` runs it SHALL run `cargo fmt --workspace --check` AND `biome check .` in addition to
  clippy (so the pre-commit hook and `just ci` actually gate formatting); the toolchain components for CI
  SHALL include `rustfmt`.
- WHEN the CI workflow references a GitHub Action it SHALL pin it to a commit SHA (`@<sha> # vX`), honoring
  the `helpers:pinGitHubActionDigests` Renovate preset already configured.
- `client/package.json` SHALL declare `"engines": { "node": ">=24.13.1 <25" }` so `.npmrc engine-strict=true`
  enforces the pinned Node; `log` SHALL be declared in `[workspace.dependencies]` and consumed via
  `log.workspace = true`.
- WHEN a contributor opens the devcontainer THE `postCreateCommand` (`just setup`) SHALL succeed — the
  container SHALL provide pinned Rust `1.96.0` (+ `wasm32-unknown-unknown`, clippy, rustfmt), pinned Node
  `24.13.1`, and `wasm-pack 0.15.0`.

**Documentation accuracy sweep (SSOT for facts)**
- THE following docs SHALL match the code: `ARCHITECTURE.md` XP formula → `bst * loser_level /
  (5 * winner_level) + 1`; the damage formula → **`u64` intermediates** in ARCHITECTURE.md + CHANGELOG +
  **ADR-0041** (whose "u32 can't overflow" rationale SHALL be corrected — `u16` power × `u16` attack *does*
  overflow `u32`); `game-core/src/combat/mod.rs` SHALL NOT claim battles are client-predicted via
  `client-wasm` (battles are server-resolved, ADR-0017).
- THE ADR catalog SHALL be navigable: ARCHITECTURE.md SHALL state that ADRs 0002–0034 live in the harness
  spec corpus and 0001/0035–0043 in `docs/adr/` (add a `docs/adr/README.md`), and SHALL list ADR-0039.
- THE CHANGELOG SHALL be regenerated (`git cliff`) through M8a; the `isWasmReady` and `frontend→client`
  named deferrals SHALL be marked resolved/closed; the M8a section SHALL note the reserved private
  `battle_wild` side-table (ADR-0045).

**Netcode & client robustness / SSOT**
- WHEN move inputs burst beyond `MOVE_QUEUE_CAP` THE SYSTEM SHALL keep the predictor consistent with
  authority — either the server acks only on successful enqueue, or the client routes the reducer rejection
  into a pending-op rollback — so a rejected move does not leave a mispredicted tile (no rubberband). A
  regression test SHALL exercise "burst > cap → reconcile".
- `PARTY_SIZE`/`PARTY_SLOT_NONE` SHALL be single-sourced from `game-core` via a `client-wasm` export
  (mirroring `step_ms()`/`move_queue_cap()`); the TS magic literals SHALL be deleted (SSOT).
- WHEN a battle overlay is active THE `KeyB` (box) toggle SHALL be suppressed; THE renderer SHALL respond
  to window `resize`; THE predictor's first drain (and each reconnect) SHALL NOT spuriously report
  `snapped: true`.

## 6. Risks / decisions
- **Sequencing vs M8b/M8c.** §1.1 touches the same `start_battle`/wild-encounter entry that M8b/M8c
  (`begin_encounter`, ADR-0045 `battle_wild`) build on. Landing M8.5 *after* M8 closes is cleanest; if it
  lands earlier, the opponent-provenance constraint MUST be re-verified against the grass-encounter caller
  (one `begin_encounter` backs both paths — M8 spec §3).
- **Opponent provenance needs an NPC model.** There is no NPC-owned-monster concept yet; the minimal fix is
  to reject non-self/non-server opponents now and let M11/M14 introduce server-authored encounter teams.
  Record the chosen mechanism in ADR-0048 (don't clamp; reject).
- **Mutation/coverage cost vs fast loop (ADR-0043).** Wiring them per-PR would fight the fast inner loop;
  the decision is a scheduled/per-milestone job + honest docs. Revisit if regressions slip through.
- **Formatter gate may surface a large first diff.** Run `cargo fmt`/`biome` once and commit the
  reformat in 5d as its own step so the gate-add PR stays reviewable.
- **CI-1 (`dtolnay/rust-toolchain@stable`)** is *not* a version-drift bug (the action reads
  `rust-toolchain.toml` when no `toolchain:` input is given); the real residue is the floating action tag
  (→ SHA-pin) and the missing `rustfmt` component (→ 5d). Don't "fix" it by pinning the channel.

### Conflict debates (winners only — full reasoning in the review report §2)
- **Divide-by-zero fix (`calc_damage`):** silent `.max(1)` clamp **lost** to *make-illegal-states-
  unrepresentable* + *defensive-code-at-boundaries-only* + *fail-loud*. Resolution → precondition +
  `debug_assert` in the pure core; reject `defense==0` at `battle_monster_from_row`; a future M14 debuff
  step clamps upstream where the debuff is applied (the M14 red-team note's own intent).
- **Opponent (side-B) HP write-back:** "add symmetric write-back" **lost** to security. On the current
  unconstrained `start_battle`, side-B write-back turns a bounded info-leak/griefing bug into persistent,
  non-consensual damage to a victim's rows. Resolution → fix authz first (5a), document the side-B no-write
  as intentional PvE (ADR-0042), guard it with the §3 "byte-for-byte unchanged" eval; symmetric write-back
  only with real PvP + per-side authority (M16/ADR-0017).

### Red-team
- A "security eval" that greps for a substring is worse than none (false confidence) → the rewritten
  `battle-reducer-security` eval must *fail* the §1.1 fixture, proving it bites.
- A doc that asserts an impossible overflow (ADR-0041 `u32`) hides the real invariant → fix the ADR, don't
  just change the prose.
- The dual-write invariant is the privacy SSOT and was only checked at struct-shape level → make it a
  mechanical parity test that fails a missing-mirror fixture.

### Simplify
No new subsystems, no schema reshape, no behavior change beyond closing the `start_battle` abuse path. The
SSOT/parity/determinism spine is confirmed working and is **not** touched except to single-source two
constants the client had duplicated.
