## 1. Problem / intent

M8.5 ("hardening-remediation") clears the bulk of the 2026-06-27 review backlog: it fixes the
`start_battle` authorization gap, the `defense==0`/`turn_number`/`as u16` rule-core contracts, the
toothless `battle-reducer-security` eval, the coverage/mutation DoD drift, the ADR-0041 `u32`→`u64`
doc error, and the tautological anchor tests. A second, independent review of the same spine
**confirmed those are the right fixes** and surfaced a small set of **verified residual gaps that M8.5
does not touch**. M8.6 closes them so the rule core is panic-free by construction and the shipped
client actually delivers the smoothness the architecture already claims.

The residuals, by ROI:

1. **Pure-core swap legality (the one M8.5 missed).** `resolve_turn`'s `Swap` branch and
   `resolve_player_swap` write a caller-supplied `team_index: u32` straight into `BattleSide.active`
   with **no bounds check and no fainted check**, then `active_monster()` indexes
   `team[active as usize]` → out-of-bounds **panic**, or a legal-but-illegal `active == fainted`
   state the resolver keeps battling. Today the only production caller (`swap_active`) bounds- and
   faint-checks first, so it is **runtime-shielded** — but the invariant lives in the *shell*, not
   `game-core` (an ADR-0003 SSOT inversion), and it is a latent panic for every future caller (M14
   deeper battle, **M16 PvP**). M8.5's rule-core slice (5b) covers `defense`/`turn_number`/
   `derive_stats` but **not** swaps; its panic-policy ADR names only skill-not-found.
2. **The M4c smoothness wiring was never connected.** `ARCHITECTURE.md` states M4c "feeds [the
   renderer] resolved positions (own from the slide clock, remote from the interpolation buffer)."
   In the shipped loop `renderEntities` feeds **raw integer tiles** (`pred.pos` for own,
   `row.tileX/Y` for remote); the `render/slideClock.ts` + `render/interpolation.ts` pure cores are
   correct and tested but have **zero importers** outside their own tests, and the store keeps
   `prev`/`latest` snapshots that nothing reads. The v1 stutter the design fights is not mitigated in
   the integrated build, and the architecture doc is inaccurate. (Distinct from M8.5f, which does
   `resize`/`lastDrainAt`/`KeyB`/burst — not the interpolation wiring.)
3. **Predictor unbounded growth + missing held-key dedup.** `Predictor.enqueue`/`#record` push to
   `#queue`/`#pending` with no cap (the cap is consulted only inside `drain`); the keydown handler
   enqueues on every event with no `e.repeat` guard and never uses the `lastQueuedDir`/`queueDepth`
   accessors the predictor exposes for exactly this dedup. Under OS key-repeat + delayed acks both
   arrays — and the per-reconcile replay cost — grow without bound. M8.5f's accept-time-ack/burst fix
   addresses the *rubberband correctness* sibling, not the client-side growth or the dedup.
4. **Cheap robustness residue.** The store leaks its live `Map`s by reference (one-way-flow hole);
   `buildBattleViewModel` guards the upper bound of `active` but not a negative; the
   `loser_base_stat_total` doc-comment says "Returns u32" but the fn returns `u16` (likely subsumed
   when M8.5b relocates the BST into game-core — verify, else fix here).

Success = the pure core cannot panic on any caller-supplied swap; the integrated client renders
own-from-slide-clock / remote-from-interpolation (matching the M4b/M4c design); the predictor is
bounded and dedups held keys; the small robustness items are closed; `just ci` green-and-meaningful.
**Pure hardening** — no player-facing behavior change except smoother motion (which the design already
specified).

## 3. Acceptance criteria (EARS)

**Pure-core swap legality (game-core; ADR-0003 SSOT; proposed ADR-0050)**
- WHEN `resolve_turn` or `resolve_player_swap` is asked to make `team_index` active and
  `team_index >= team.len()` THE SYSTEM SHALL reject the swap (no state mutation, no `Switch` event;
  a divergence/`Rejected` signal is acceptable) and SHALL NOT panic-index.
- WHEN a swap targets a **fainted** team member THE SYSTEM SHALL reject it the same way (preserve the
  "active is always conscious" invariant the auto-switch path already upholds via
  `next_conscious_index()`).
- THE `BattleSide.active` field SHALL be settable only through a checked operation (e.g.
  `BattleSide::set_active(idx) -> Result<(), _>`, or a private field with a constructor/setter that
  validates), so an out-of-range or fainted active is **unrepresentable**, not merely unreached;
  direct `state.side_x.active = idx` writes SHALL be removed from the resolver.
- THE `swap_active` server reducer SHALL keep its bounds/faint/identity checks as **defense-in-depth**
  (reject-not-clamp) and SHALL NOT rely on the core for authorization.
- PROOF-OF-TEETH: a game-core test SHALL call `resolve_turn`/`resolve_player_swap` with (a) an
  out-of-bounds `team_index` and (b) a fainted target, asserting **no panic** and an unchanged, legal
  state — and SHALL fail if the checked setter is reverted to a raw assignment.

**Render smoothness wiring (client; ARCHITECTURE M4b/M4c)**
- WHEN the render loop resolves the **own** character's position THE SYSTEM SHALL drive it from the
  self-owned `SlideClock` (target set on each drain tile-change, snapped on `DrainResult.snapped`),
  not the raw predicted integer tile.
- WHEN the render loop resolves a **remote** character's position THE SYSTEM SHALL read the
  interpolation buffer at `now − interpDelay` between the two bracketing snapshots
  (hold-not-extrapolate), using the `prev`/`latest` snapshots the store already maintains.
- THE unused `prev`-snapshot bookkeeping SHALL either be consumed by the interpolation path or removed
  (no dead weight left behind).
- PROOF-OF-TEETH: a headless/integration test (or a `window.__game()` snapshot assertion in the e2e
  golden flow) SHALL show rendered positions taking **fractional** inter-tile values during motion;
  a renderer fed raw integer tiles SHALL fail it. The existing `render/slideClock.test.ts` /
  `render/interpolation.test.ts` pure-core proofs SHALL remain green.

**Predictor flow-control (client; ADR-0013)**
- WHEN keydown fires with `event.repeat == true` THE SYSTEM SHALL ignore it (no enqueue).
- WHEN the next intended direction equals `predictor.lastQueuedDir` THE SYSTEM SHALL NOT enqueue a
  duplicate held move.
- WHEN `enqueue` is called with `#queue.length >= queueCap` THE SYSTEM SHALL coalesce/drop to keep
  `#queue` bounded (mirroring the server's `MOVE_QUEUE_CAP` semantics), and `#pending` SHALL not grow
  without bound across un-acked bursts.
- A regression test SHALL drive a held-key burst with delayed acks and assert bounded array sizes and
  a correct post-reconcile tile (coordinate with M8.5f's burst→reconcile test; do not duplicate it).

**Robustness residue (client + server-module comment)**
- `AuthoritativeStore.speciesMap()` / `skillMap()` SHALL return a defensive copy or a frozen
  read-only view, never the live private `Map` (uphold the one-way-flow invariant the store header
  asserts).
- `buildBattleViewModel` SHALL fail soft (return `null`) on a negative `active`, not only on
  `active >= team.len()`.
- THE `loser_base_stat_total` doc-comment SHALL match its signature (it returns `u16`) — OR, if
  M8.5b has relocated the BST into a pure game-core fn, this item is **subsumed**; verify and close.  **[CLOSED — SUBSUMED by M8.5b / ADR-0049 §4 ("BST — owned by the rule layer"); verified 2026-06-27. Doc-comment accurate; m7b + base_stat_total gating tests bind the contract. No code change.]**

## 7. Review / red-team notes
### Source
The 2026-06-27 read-only multi-lens review (server-security, client-TS, test-integrity/gate-wiring,
game-core math). Cross-referenced against `M8.5-hardening-remediation.spec.md` §3; M8.6 keeps **only**
the verified findings M8.5's acceptance criteria do not cover (swap legality, smoothness wiring,
predictor bound/dedup, three Low-severity polish items).

### Calibration (where the two reviews differ)
- `start_battle` arbitrary `opponent_identity`: the second review initially **cleared** it (side-B
  rows are not written back; `monster_pub` derived stats are already public, so no row-corruption and
  no *net-new* info leak) but **under-weighted XP fabrication** (conscripting any high-BST monster to
  farm scaled XP) and the server-authority/consent principle. **M8.5a is correct to make it P0**;
  M8.6 defers entirely to it.

### Red-team
- A pure function that panics on caller input is a contract bug even when every *current* caller
  guards it — the guard is one refactor (or one new caller, e.g. PvP) away from being removed. Make
  the illegal state unrepresentable in the type, not merely unreached by convention.
- A render layer that is fully tested in isolation but imported by nothing is "green but dead" — the
  proof-of-teeth must assert the *integrated* path produces fractional motion, or the gate is theatre.
