## 1. Problem / intent

M8.5 + M8.6 clear the first two reviews' backlog (the `start_battle` authz gap, the rule-core `defense==0`/`turn_number`/`as u16` contracts, swap legality, the toothless `battle-reducer-security` eval, the M4c smoothness wiring, predictor flow-control, the coverage/mutation DoD drift, the doc sweep). A **third**, independent review of the same spine — this time on a snapshot that *includes the M8b/M8c/M8d merges both prior reviews predated* (private encounter table, grass-encounter spine, recruit-by-weaken + inventory) — confirmed the architecture remains sound and surfaced a small set of **verified residual gaps M8.5/M8.6 do not address**, concentrated in the newly-merged code and in two load-bearing CI gates.

The residuals, by ROI:

1. **The schema-snapshot gate guards only ONE of ~ten tables.** `evals/battle-schema-snapshot.eval.mjs` parses *all* table structs but only ever asserts `tables['battle']` against a single `battle-columns.json` baseline. The M8b–M8d tables — `encounter`, `battle_wild`, `inventory`, `item_row` — plus `monster`/`monster_pub` have **no** non-additive-change guard. A dropped column / changed PK / changed type on any of them passes CI green. ADR-0006 (the project's #1 deferred risk — schema migration) requires "a schema-snapshot eval flags non-additive diffs"; it does so for 10% of the schema. (High.)
2. **A false privacy guarantee in the `inventory` model** (3-lens consensus). The table doc, a red-team test comment, and the `inventory-privacy` eval header all assert "RLS by `owner_identity`" / "a client sees its OWN items," but `client_visibility_filter` appears **zero** times repo-wide (ADR-0040 records it as unenforced in the pinned toolchain) and ADR-0046 deliberately chose `public`, no-RLS. The data is low-stakes today, but the comment will mislead a future contributor into shipping a higher-stakes column into a table they believe is owner-scoped — or skipping the flagged M16 RLS pass.
3. **Dev/test reducers ship client-callable in release builds.** `start_wild_battle` and `grant_bait` are plain `#[reducer]`s (no `#[cfg]`/feature gate). `start_wild_battle` additionally trusts a client-supplied `zone_id` (its own dev-note only addresses the *seed* cheat) → a client rolls any zone's private encounter table from anywhere (rare-encounter-table spoof); `grant_bait` mints bait into the caller's own inventory unbounded (the 99/call cap is cosmetic — callable in a loop). Both are acknowledged "remove at M9" debt, but nothing *mechanically* keeps them out of a production deploy.
4. **The render loop never shows a battle's outcome.** `refreshBattle()` sources the overlay solely from `store.ongoingBattle()` (filters `outcome == Ongoing`), so a resolved battle vanishes instantly — the `#renderOutcome` ("Victory!/Defeat…/Got away safely!") and recruit-success frames are dead code in the integrated build.
5. **Two gate/test-integrity residues.** The `zoned-schema` eval only inspects tile-bearing tables, so the per-zone `encounter` table (and any future non-tile per-zone table) escapes ADR-0007's "every world table carries an indexed `zone_id`" gate; and the `wild_iv_recoverable_from_public_derived_stats` red-team test is a **self-oracle tautology** — its hardcoded `public_stat_hp = 26` is unreachable (the real HP range is 29–32), so it always passes through a fallback that recomputes the expected value with the same function under test, proving nothing about the ADR-0045 inversion channel it claims to demonstrate.
6. **The recruit-by-weaken *bait* path is unreachable on the client (DECISION).** M8d shipped the server reducers + inventory, but the client never subscribes to `inventory`/`item_row` and never passes `baitItems` to `buildBattleViewModel`, so the bait `<select>` is always "No bait" and the e2e (`recruit.spec.ts`) is `test.fixme`. The feature is half-delivered and the gap is untracked.
7. **Cheap robustness/contract polish** — see §3 low items.

Success = the schema-snapshot gate covers every table; the privacy docs match the code; the dev reducers cannot ship in release; the player sees battle outcomes; the two CI gates bite on their real targets; the small contract items are closed; `just ci` green-and-meaningful. **Pure hardening** — no new game-design surface; the only player-visible change is the (already-intended) battle-outcome frame, and — if §6's DECISION is taken — the bait UI.

## 2. Scope

**In scope** (grouped; criteria in §3, slice/`touches:` map in §4)
- **Schema-snapshot gate generalization** — snapshot every `#[spacetimedb::table]` struct (columns + PK) to a per-table baseline; fail any removed column / changed PK / changed declared type; cover `encounter`, `battle_wild`, `inventory`, `item_row`, `monster`, `monster_pub` at minimum. Proof-of-teeth on a non-`battle` table.
- **Zoned-schema gate broadening** — flag any per-zone table whose `zone_id`/`map_id` is neither `#[primary_key]` nor `#[index(btree)]`, not only tile-bearing tables; keep the existing tile teeth and add an `encounter`-shaped teeth fixture.
- **Dev/test-reducer release-gating** — put `start_wild_battle` + `grant_bait` behind a `#[cfg(feature = "dev_reducers")]` (off in release/publish); if `start_wild_battle` is retained at all, derive the zone from the caller's `Character.zone_id` (or reject on mismatch — reject-not-clamp), never trust the client `zone_id`. Proposed ADR-0051.
- **Inventory structural + doc integrity** — add a DB-level unique index on `(owner_identity, item_id)` so the "one stack per item" invariant is structural, not convention; correct the false "RLS by owner_identity" claim in the table doc, the `inventory-privacy` eval header, and the `redteam_m8d` comment to match ADR-0046 (public, no transport RLS).
- **Battle-outcome rendering** — feed the overlay from the player's most-recent battle (not Ongoing-only) so a resolved outcome renders once, with an explicit dismiss (Escape / brief timeout).
- **Test/contract polish** — fix the IV-inversion red-team test to a reachable HP + a real candidate-set assertion (drop the self-oracle fallback); strengthen `recruit-reducer-security` ownership/wild guards from substring-presence to a rejecting-comparison check + a "tokens-present-but-no-rejection" bad fixture; add the `recruit_chance` input `debug_assert` its consumer `attempt_recruit` already has; guard `roll_encounter`'s `total_weight` sum against `u32` overflow (checked fold or a per-zone weight cap in `validate_encounters`); wire or remove the decorative `Config.content_version`; correct ADR-0044's `species_id: u16` → `u32`.

**Out of scope (named deferrals — declared, not dropped)**
- **Everything M8.5 / M8.6 own** — `start_battle` authz, rule-core `defense==0`/`turn_number`/`derive_stats`/swap legality, the `battle-reducer-security`/dual-write eval teeth + anchor tests + coverage/mutation, fmt/biome + SHA-pin + devcontainer + `log`, the ADR/CHANGELOG doc sweep, accept-time-ack + party-constant SSOT, M4c smoothness wiring, predictor flow-control, store map-leak, negative-`active` guard. M8.7 must not duplicate or re-litigate these.
- **Symmetric side-B HP write-back** → **M16** (reaffirmed). **Battle reaper for abandoned `Ongoing` battles** → **M20** (reaffirmed). **Client a11y** → **M23**.
- **IV-inversion *mitigation*** (perturb/quantize public derived stats) → not now; ADR-0045 accepts the channel. M8.7 only fixes the *test* that pretends to prove it.
- **Per-owner transport RLS on `inventory`** → **M16** (ADR-0046 residual a). M8.7 only corrects the doc and makes the single-stack invariant structural.
- No new content, no schema reshape beyond additive indexes/columns/feature-gates.

## 3. Acceptance criteria (EARS)

**Schema-snapshot gate (evals; ADR-0006)**
- WHEN `battle-schema-snapshot` (or its successor) runs THE SYSTEM SHALL assert column-set + PK stability for **every** `#[spacetimedb::table]` struct against a per-table baseline, and SHALL fail a fixture that removes a column / changes a PK / changes a declared field type on a **non-`battle`** table (proof-of-teeth on `inventory` or `encounter`).
- THE baseline SHALL include `encounter`, `battle_wild`, `inventory`, `item_row`, `monster`, `monster_pub` (and `battle`, `character`, `player`, content tables) so a future non-additive change to any is caught at CI, not deploy.

**Zoned-schema gate (evals; ADR-0007)**
- WHEN the zoned-schema eval scans the module THE SYSTEM SHALL flag any table that declares a `zone_id`/`map_id` field which is neither `#[primary_key]` nor `#[index(btree)]` — not only tables with `tile_x`/`tile_y` — and SHALL pass the `encounter` table (zone_id as PK) while still flagging the existing zoneless-spatial teeth fixture.

**Dev/test-reducer gating (server-module; proposed ADR-0051)**
- WHEN the module is built for release/publish THE SYSTEM SHALL NOT expose `start_wild_battle` or `grant_bait` as callable reducers (gate behind `#[cfg(feature = "dev_reducers")]`, off by default; CI/dev may enable it for the e2e/golden flow).
- IF `start_wild_battle` is retained behind the gate THE SYSTEM SHALL derive the encounter zone from the caller's `Character.zone_id` (or reject with `Err` when a passed `zone_id` ≠ the character's) — it SHALL NOT roll an arbitrary client-named zone's encounter table (reject-not-clamp).

**Inventory integrity (server-module; ADR-0046)**
- THE `inventory` table SHALL enforce at most one row per `(owner_identity, item_id)` via a DB-level unique constraint/index (additive — ADR-0006-safe), so a duplicate stack is structurally impossible, not merely avoided by find-then-update convention. `grant_item`/`consume_one` logic is unchanged.
- THE `inventory` table doc, the `inventory-privacy.eval.mjs` header, and the `redteam_m8d_tests.rs` comment SHALL state the real posture (public / world-readable counts / no transport RLS; owner-scoping is a client subscription filter only; M16 RLS tracked) — no text SHALL claim an `owner_identity` RLS filter that does not exist.

**Battle-outcome rendering (client; ADR-0014/0042)**
- WHEN a player's battle resolves (`SideAWins`/`SideBWins`/`Fled`/recruit-success) THE SYSTEM SHALL render the terminal outcome frame at least once (the existing `#renderOutcome` path) rather than hiding the overlay the instant `outcome != Ongoing`, and SHALL provide an explicit dismiss (Escape and/or a brief timeout). The Ongoing-only auto-show behavior for active battles is preserved.

**Test/contract polish (game-core + evals)**
- THE `wild_iv_recoverable_from_public_derived_stats` test SHALL use a `public_stat_hp` value actually produced by `derive_stats` for a known IV, assert the brute-force search recovers a **small candidate set containing that IV**, and SHALL NOT contain a fallback branch that recomputes the expected value with the function under test (no self-oracle). It SHALL fail if the inversion claim is false.
- THE `recruit-reducer-security` ownership/wild-battle checks SHALL require a *rejecting comparison* (e.g. `player_identity != ctx.sender` followed by an early `Err`), not mere substring presence, and SHALL fail a fixture where the tokens appear but no rejection is wired.
- THE `recruit_chance` pure fn SHALL `debug_assert!(base_rate <= 1000 && bait_bonus <= 1000, …)`, mirroring `attempt_recruit`'s precondition (fail-loud parity; no release behavior change).
- WHEN `roll_encounter` sums entry weights THE SYSTEM SHALL not overflow `u32` (checked/saturating fold returning `None`, or a per-zone aggregate-weight ceiling enforced by `validate_encounters`).
- THE `Config.content_version` field SHALL either be incremented by `sync_content` (and read by re-derive/cache logic) or be removed until a consumer exists (no decorative version column).
- ADR-0044 SHALL state `species_id: u32` (matching `EncounterEntryRow`/the species registry width).

### Calibration (where this review differs from the prior two)
- The first two reviews swept M0–M8a; they could not have seen `start_wild_battle`/`grant_bait`/`inventory`/`encounter`/`battle_wild` (M8b–M8d). This review's net-new value is almost entirely in that delta + the gates that silently failed to grow with it.
- The `inventory` "RLS by owner_identity" claim was independently flagged by **three** lenses (server-security, schema, spec/doc) → high confidence it is a real, load-bearing doc error, not a nit. It is a *documentation* bug (the behavior matches ADR-0046); the danger is a future contributor trusting the comment.

### Red-team
- A schema-migration gate that guards one table on a project whose **#1 deferred risk is schema migration** (PLAN.md G1) is worse than none — it advertises a safety net that covers 10% of the schema. Generalize it before M9 adds more tables.
- A red-team test whose "exploit proof" can only pass through a branch that recomputes the expected value with the code under test proves determinism, not the exploit. Make the oracle independent or the gate is theatre (the same principle M8.6 applied to the dead render layer).
- A dev/cheat reducer that is "documented to remove at M9" but has no build-time gate **is** a production reducer until someone remembers. Make the removal mechanical (`#[cfg]`), not a memory.

### Simplify
No new subsystems, no schema reshape, no behavior change except the (already-intended) battle-outcome frame and the optional bait UI. Every other item is a gate generalization, a feature-gate, an additive index, a `debug_assert`, or a doc correction. The SSOT/parity/determinism/privacy-split spine is confirmed working on the M8b–M8d code and is not touched.
