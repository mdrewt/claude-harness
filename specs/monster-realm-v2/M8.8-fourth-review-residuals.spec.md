## 1. Problem / intent

M8.5–M8.7 clear the first three reviews' backlog (start_battle authz; rule-core `defense==0`/`turn_number`/`as u16`/swap legality; eval/dual-write teeth + coverage/mutation; render-smoothness wiring; predictor flow-control/dedup/bound; dev-reducer release-gating; schema-snapshot + zoned-schema gate generalization; the inventory RLS doc; battle-outcome render; the IV-inversion self-oracle test; the doc sweep). A **fourth**, independent review of the same spine — on a snapshot that includes M8.6a — confirms the architecture remains sound and the SSOT/parity/privacy-split/determinism spine holds. It surfaced a small set of **verified residual gaps none of M8.5/M8.6/M8.7 address**, concentrated in two places the prior reviews under-weighted: the **build/lint enforcement of the determinism invariant** (the gate has escape hatches), and the **netcode test substrate** (the sim-harness asserts far less than its role claims).

The residuals, by ROI:

1. **Release builds silently wrap on integer overflow (4-lens consensus root-cause).** The workspace `Cargo.toml` defines no `[profile.release] overflow-checks` (verified: no `[profile.*]` anywhere, no `.cargo/config.toml`). For a project whose thesis is determinism + fail-loud, the shipped artifacts — the wasm prediction pkg, the SpacetimeDB `cdylib`, benches — all build in release and **wrap** instead of panicking. Debug/test runs (checks on) and release (checks off) diverge: an overflow that tests catch silently corrupts in production. This is the amplifier that turns every "latent overflow" into a real release defect (e.g. residual 2). (High.)
2. **The recruit-fail path advances `turn_number` out-of-band, bypassing the SSOT turn-limit terminal.** `attempt_recruit` does `battle.state.turn_number += 1` directly in the reducer (`server-module/src/lib.rs:2034`), then calls `resolve_enemy_turn` — neither honors the `u16::MAX → Fled` terminal that `resolve_turn` owns (`resolve.rs:165-170`). A code comment at `resolve.rs:163-164` already admits the gap ("this guards `resolve_turn` only; the `attempt_recruit` reducer advances `turn_number` out-of-band"). A battle that never faints (e.g. a skill-less wild) driven to `u16::MAX` turns **panics in debug / wraps in release** (with residual 1) instead of terminating — and it's an ADR-0003 SSOT inversion (turn-advance is a rule that lives in two places). (Medium.)
3. **The determinism lint gate has coverage holes.** `clippy.toml`'s `disallowed-methods` bans `std::time::{SystemTime,Instant}::{now,elapsed}`, `rand::thread_rng`, `rand::random` — but NOT `rand::rngs::OsRng`, `getrandom::getrandom`/`fill`, `rand::rng` (the rand 0.9 alias), or `chrono::Utc::now`/`Local::now`. All of these entropy/wall-clock sinks are in the resolved graph (`Cargo.lock`: `chrono 0.4.45`, `rand 0.9.4`, `getrandom`, `fastrand 2.4.1`), reachable from the determinism-critical crates. A clean `just lint` therefore gives false confidence; a contributor can read a wall clock or OS entropy in a "pure" rule path and pass the gate. (High — it undercuts the project's #1 mechanical invariant.)
4. **The sim-harness "assert convergence" is theatre.** `Link::transport` (latency/loss/reorder) and the authoritative `ServerWorld` are **disjoint**: the only consumer of `transport` (`sim-harness/src/bin/netcode_check.rs`) asserts only link determinism (`a == b`) and `0 < delivered < sent`; `ServerWorld` is referenced nowhere outside its own `#[cfg(test)]`. The headline netcode property — "under latency/loss/reorder, two clients still converge, no desync" (ADR-0013) — is asserted **nowhere**, on the very substrate the plan says M1+ netcode tests build on. Additionally: `reorder` is claimed but never asserted to occur, and `forfeit-on-disconnect`/`turn-deadline` are claimed harness invariants that have **zero** implementation or eval (only future-M16 prose). (High value; the convergence driver is the only Medium-effort item.)
5. **Client prediction robustness gaps (post-reconnect & divergence).** (a) On reconnect, `main.ts` builds a fresh `Predictor` whose `#nextSeq` restarts at 0, while the server's persisted `player.last_input_seq` (u64) is far higher; `reconcile`'s `p.seq > ackedSeq` filter then **drops every freshly-recorded post-reconnect intent** until the local counter climbs back past the server's value — the player appears frozen after a reconnect (no re-seed exists). (b) `reconcile`'s documented divergence-return is **discarded** at `main.ts:85`, and there is no `keyup`/held-key tracking, so after a genuine server pullback the held movement key is never re-issued (stall until re-press) — the exact mechanism the boolean was built for. Distinct from M8.6c's `e.repeat`/dedup/bound (which does not touch reconnect-seq or the divergence return). (a) High; (b) Medium. **SERIAL after M8.6c** (shares `main.ts`/`predictor.ts`).
6. **Content invariant: `skill.accuracy` is unvalidated.** `validate_content` enforces `power > 0` but never checks `accuracy`; `accuracy` is a `u8` with no range invariant. Since `accuracy_check` is `roll < accuracy` with `roll ∈ 0..=99`, a content typo of `accuracy: 0` makes a skill **always miss** forever (no load error), and `accuracy ≥ 100` makes it always hit — an illegal-but-representable state, the same class as the enforced `power == 0` check. (Medium — cheap, real footgun.)
7. **Gate/test-integrity polish (Low).** `evals/e2e-desync-teeth.eval.mjs` is a substring proxy that passes on textual presence and cannot detect a present-but-disabled e2e job (`if: false`/`continue-on-error`); the parked `#[ignore]` owner-change spec-gap test (`m7b_redteam_tests.rs:152`) has no mechanism forcing its revival when the M11+ trade reducer lands; the level-up HP-heal delta is computed in the reducer (`lib.rs:1850-1852`), not `game-core` (a minor SSOT residual, sibling to residual 2).

Success = release builds fail loud on overflow; the determinism gate covers every entropy/clock sink it claims to (with proof-of-teeth); the turn-limit terminal is owned once in `game-core` and honored on the recruit path; the sim-harness actually proves convergence under loss/reorder; the client survives a reconnect and re-issues a held key on divergence; content validation rejects an unusable `accuracy`; the two gate residues are closed; `just ci` green-and-meaningful. **Pure hardening** — the only behavior-visible changes are smoother post-reconnect/divergence movement (already-intended by ADR-0013) and loud-instead-of-silent failure on malformed content/overflow.

## 3. Acceptance criteria (EARS)

**Release fail-loud + determinism gate (build/lint; ADR-0003/0010; proposed ADR-0054)**
- WHEN the workspace is built in release THE SYSTEM SHALL have `overflow-checks = true` for the `release` (and `bench`) profile, so integer overflow in `game-core`/`server-module` panics (fails loud) rather than wrapping. A test/CI step SHALL demonstrate a release build aborts on a deliberate overflow fixture (proof-of-teeth), and the existing suite SHALL stay green under release checks.
- WHEN `just lint` runs THE SYSTEM SHALL reject (`-D warnings`) a use of `rand::rngs::OsRng`, `rand::rngs::ThreadRng`, `getrandom::getrandom`, `getrandom::fill`, `rand::rng`, `chrono::Utc::now`, or `chrono::Local::now` in any workspace crate — not only `std::time::*`/`rand::thread_rng`/`rand::random`. A proof-of-teeth fixture SHALL prove the gate bites on at least one OS-entropy path and one `chrono` wall-clock path.

**Recruit-path turn terminal (game-core + server-module; ADR-0003)**
- THE turn-number advance with its `u16::MAX → Fled` terminal SHALL be owned by a single `game-core` operation; `attempt_recruit` SHALL advance the turn only through it, never via a raw `battle.state.turn_number += 1` in the reducer.
- WHEN `attempt_recruit` (failure branch) advances a battle already at `turn_number == u16::MAX` THE SYSTEM SHALL set `outcome = Fled` (terminate) rather than panic (debug) or wrap to 0 (release). PROOF-OF-TEETH: a test drives a wild battle to `u16::MAX`, calls the recruit-fail path, and asserts `outcome == Fled` with no panic and `turn_number` unchanged; it SHALL fail if the reducer reverts to a raw in-shell increment.
- THE level-up HP-heal-from-growth-delta SHALL be computed by a `game-core` function (returned alongside the level-up result), not re-implemented in the reducer.

**Sim-harness convergence (sim-harness + evals; ADR-0013/0009/0010)**
- WHEN ≥2 simulated clients' intent streams are passed through `Link::transport` with latency + jitter + loss THE SYSTEM SHALL apply the delivered (dropped/reordered) stream to `ServerWorld` and assert both clients' final authoritative state is identical (convergence), deterministically across multiple seeds. A known-bad order-dependent fixture SHALL be rejected (proof-of-teeth), and a new/extended eval SHALL gate this in `just eval`.
- THE harness SHALL assert that reorder actually occurs (delivered order ≠ send order) for the standard jittered config, so a `jitter: 0` regression is caught.
- THE sim-harness's stated invariants SHALL match what it tests: either forfeit-on-disconnect + turn-deadline are implemented in `ServerWorld` with assertions + an eval, OR the harness role description (and any snapshot/spec text) SHALL scope them to M16 so the claim is not load-bearing.

**Client prediction robustness (client; ADR-0012/0013)**
- WHEN the client reconnects THE SYSTEM SHALL seed the new predictor's next sequence from the authoritative `last_input_seq` so the first post-reconnect intent's seq is strictly greater than the server's last ack and survives `reconcile`. A regression test SHALL reconnect with a non-zero acked seq and assert the first post-reconnect enqueue is not dropped.
- WHEN `reconcile` returns a genuine divergence THE SYSTEM SHALL re-issue the currently-held movement direction (tracking held keys via a `keyup` handler), so motion does not stall until the user re-presses. A regression test SHALL assert a held key resumes after a simulated server pullback.
- THE `last_input_seq` u64→number conversion SHALL be documented/bounded consistent with the `convert.ts` `moveStartedAtMs` precedent (comment + assertion, or keep seqs `bigint` end-to-end).

**Content accuracy validation (game-core; content-as-data)**
- WHEN `validate_content` runs THE SYSTEM SHALL return `Err` for any skill with `accuracy == 0` or `accuracy > 100`, so an unusable (always-miss) or illegal (always-hit) skill cannot load. PROOF-OF-TEETH: a fixture with `accuracy: 0` (and one with `accuracy: 101`) SHALL be rejected; the existing valid content SHALL still load.

**Gate/test integrity (evals + game-core)**
- THE `e2e-desync-teeth` eval SHALL fail when the e2e CI job is present but disabled (`if: false`, `continue-on-error: true`, or a non-PR trigger), not only when the job text is removed — i.e. parse the job structure, not substrings.
- THE parked owner-change spec-gap test SHALL be backed by a mechanism (eval or test) that fails CI once a `trade`/`transfer`/ownership-change reducer exists in `server-module` while the test is still `#[ignore]`d — so the spec gap cannot silently outlive its precondition.
