## 1. Problem / intent

M8.9 removes the **two structural collision points** that force server- and content-work to serialize under the `touches:`-disjoint fan-out model: **(A) the `server-module/src/lib.rs` monolith** (a pure code reorg, ADR-0055) and **(B) the monolithic per-registry content files** (a glob loader, ADR-0056). They are **independent workstreams** (disjoint `touches:` — `server-module/**` vs `game-core/content/**` + the loader) and can themselves fan out against each other.

### (A) The server-module monolith

`server-module/src/lib.rs` is a single flat module of **~2081 production lines** (3243 total incl. ~1160 inline test lines): **15 `#[table]` structs, 18 `#[reducer]`s, and 27 helper fns** in one namespace, spanning lifecycle, content-sync, movement, monster-management, battle, and taming. Every other crate is healthy — the client's largest production file is `main.ts` at 310 lines (well-split across `render/net/ui/prediction/convert`), and game-core's production code per file is ≤ ~520 lines (its large *totals* are inline `#[cfg(test)]` modules, not logic). So this is not a project-wide problem; it is one monolith.

Why it matters here specifically — **it throttles the build fleet's parallelism.** Per `PLAN.md` §9 (and `WORKSPACE-PLAN.md` §7), the supervisor runs two slices concurrently **only when their `touches:` sets are disjoint**; a slice that lacks a `touches:` declaration, or that names `server-module/src/lib.rs`, collides with every other `lib.rs` slice and is forced **serial**. Because *all* server-side gameplay (battle, taming, movement, content, monster-mgmt) lives in that one file, every server-side slice declares `touches: server-module/src/lib.rs` and therefore **cannot fan out** — exactly the bottleneck observed across M7/M8/M8.5–M8.8, where server slices chained serially. Splitting the file by domain turns `touches: server-module/src/lib.rs` into `touches: server-module/src/battle.rs` (or `taming.rs`, `movement.rs`, …), so genuinely independent server slices (e.g. a battle-systems slice for M14 and a trading slice for M15) become **disjoint and parallelizable**. Secondary wins: faster review, smaller diffs, and tighter impact analysis (a change to taming no longer shows a 2000-line file in the blast radius).

The split is low-risk because the blast radius is internal: **nothing in the workspace depends on `server-module`** (it is the cdylib leaf — verified: it is a workspace member but no crate imports it), and table/reducer **names are explicit** (`name = character`, etc.), so a behavior-preserving move regenerates **byte-identical** TypeScript bindings. The existing `bindings-drift` and `schema-snapshot` evals therefore *prove* the reorg changed no observable surface — they are the milestone's gate.

### (B) The content-registry monoliths

The six content registries are each **one monolithic RON file** (`game-core/content/{species,skills,items,encounters,zones,type_chart}.ron`), embedded at compile time via `include_str!`. `PLAN.md` §9 names pure **content-data additions** a *natural leaf* for parallelism — yet two slices adding species (or items, encounters, zones) both edit the one `species.ron` and **collide**. Content is tiny today (≤ ~2 KB/file) but Phase B/C is content-heavy: **M9** (training items), **M10** (fusion recipes), **M11** (multi-zone world), **M12** (dialogue/quests), **M13** (shop items), **M14** (abilities) all add content. Making each registry a **glob-loaded directory** — a `build.rs` embeds every `content/<registry>/*.ron` in sorted order and the loader concatenates the parsed `Vec`s — turns `touches: …/species.ron` into `touches: …/species/<new>.ron`, so content additions become **disjoint files**. This is behavior-preserving (the merged registry is byte-for-byte the same set of rows), but unlike (A) it **adds a `build.rs`** (no runtime dependency) and therefore carries its own ADR (0056) and a **content-parity** proof rather than the bindings proof — content does not affect `module_bindings` (those derive from server-module tables/reducers, not data) nor the schema snapshot. (Done now because M9/M10 add content immediately — not speculative — and doing it before the Phase-B content explosion avoids a larger migration later.)

## 2. Scope

**In scope** (grouped; criteria in §3, slice/`touches:` map in §4)
- **Module-tree split of `server-module/src/lib.rs`** into cohesive submodules of the same crate (see §4 for the layout), preserving every table/reducer name, signature, and ordering-independent behavior. `lib.rs` shrinks to module wiring + the lifecycle reducers (`init`/`sync_content`/`on_disconnect`).
- **Consolidate the reducer ownership-check boilerplate.** 52 `log_reject` call-sites and a repeated `owner != ctx.sender → log_reject → Err` preamble across 18 reducers collapse into a single `require_owner(ctx, reducer, owner) -> Result<(), String>` (generalizing the existing `authorize_move`), in a `guards` module. Pure de-dup; identical reject behavior.
- **Extract `server-module`'s inline `#[cfg(test)]` tests** into sibling test modules (matching the convention game-core already uses: `pub mod *_tests;`), so the production modules stay lean.
- **Verify-first spike (gating):** confirm `#[spacetimedb::table]` / `#[reducer]` register correctly when the item lives in a **submodule** (not the crate root) on the pinned `spacetime 2.6.0` — the one load-bearing assumption (see §6 fallback).
- **ADR** for the new module boundary (harness **ADR-0055**; project-side filed at build), per `standards/adr-process.md`.
- **(B) Content as glob-loaded directories.** Add a `game-core/build.rs` that embeds every `content/<registry>/*.ron` (sorted → deterministic) and have the loaders in `content.rs` concatenate the parsed `Vec`s; migrate each registry file into its directory (`species.ron` → `species/000-core.ron`, …) with **identical rows / ids**. After this, adding content is a new file in the registry dir — no `content.rs`/loader edit. Registries that grow in parallel get directories (`species`, `skills`, `items`, `encounters`, `zones`, + future `fusions`/`dialogue`/`quests`); `type_chart` may stay a single file (one coherent matrix, rarely appended in parallel). **ADR-0056** (content-directory glob loading; adds a `build.rs`, no runtime dep).

**Out of scope (named deferrals — declared, not dropped)**
- **The client and game-core *rule* code** — already appropriately sized; not restructured (`main.ts` 310 lines; game-core rule files ≤ ~520). *Exception:* workstream (B) touches `game-core/src/content.rs` (loaders), adds `game-core/build.rs`, and reorganizes `game-core/content/**` — that is the content workstream, not a rule-logic change.
- **Integration-runtime isolation** (per-slice SpacetimeDB db-name / port / instance, so two `e2e`/`publish` slices don't collide on the fixed `monster-realm` db + `--delete-data` + `:3000` + Playwright `:5290`) → **separate harness/test-infra work**, not a behavior-preserving code reorg: it changes how publish/e2e *run* and lives partly in the project's `justfile`/`playwright.config`/e2e setup + the supervisor's fan-out *resource* rules (today gated on files + RAM, not runtime resources). Track before the integration-heavy phase (M11 e2e, M16 PvP, M18 raids). Out of M8.9.
- **Pre-splitting shared type files** (`combat/types.rs`, `monster/types.rs`) ahead of the milestones that extend them → **YAGNI; the extending milestone splits them** (M14 as it adds status/abilities/weather; M9/M10 as they add progression/evolution). The "don't grow a new monolith — split as part of the milestone" rule is already in the `spec-kit` skill + the M11–M25 fan-out notes. Out of M8.9 (speculative seams).
- **`sccache`-by-default for parallel worktrees** → harness build-infra (the supervisor already uses it *if installed* + checks RAM headroom; provisioning it by default is a devcontainer change). Out of M8.9.
- **Relocating reducer-resident *rules* into `game-core`** (the level-up HP-heal at `lib.rs:1850`; the recruit-path `turn_number` advance at `lib.rs:2034`) — these are **SSOT/correctness** items already owned by **M8.8** (8b/8f). M8.9 is a *layout* reorg and must not duplicate or pre-empt them; it inherits whatever shape M8.8 leaves. If M8.9 follows M8.8 (recommended), those rules already live in `game-core` and M8.9 simply places the thin call-sites in the right module.
- **`game-core` inline-test extraction** (`resolve.rs` 913 inline test lines, `damage.rs` 513, `monster/types.rs` 392, etc.) → **DECISION (§6)**: a real but lower-value consistency cleanup that *collides* with M8.7a/M8.8 test edits on `game-core/src/combat/`. Default: **defer** to a separate, later housekeeping slice once the combat-test churn settles; do **not** bundle it here (it would re-serialize against the very combat work this milestone is meant to unblock).
- **Any schema change, new table/column/index, behavior change, or new game-design surface.** If a diff changes the schema snapshot or the generated bindings, it is a bug in the reorg, not an intended change.
- **Splitting `game-core` *crates* further** (e.g. a separate `combat` crate) — YAGNI; the module boundaries inside `game-core` are already fine.

## 3. Acceptance criteria (EARS)

**Behavior preservation (the gate; ADR-0009/0010)**
- WHEN the module is rebuilt and `just gen` regenerates the TypeScript bindings after the split THE SYSTEM SHALL produce **byte-identical** `client/src/module_bindings/` output (the `bindings-drift` eval SHALL pass with zero diff).
- WHEN the `schema-snapshot` eval runs after the split THE SYSTEM SHALL report **no** column/PK/type change for any table (the reorg is schema-neutral).
- THE full `just ci` (lint + typecheck + test + eval + security + wasm + client-typecheck + client-test) SHALL be green, and the existing server-module test suite SHALL pass **unchanged in assertions** (tests move files but keep their bodies).

**Module structure (ADR-0005; proposed ADR-0055)**
- THE `server-module` crate SHALL expose its tables and reducers from **cohesive submodules by domain** (schema, guards, marshal, content, movement, monster-management, battle, taming), with `lib.rs` reduced to module declarations + the lifecycle reducers, such that no single `server-module` source file exceeds roughly ~400 production lines.
- EACH future server-side slice SHALL be able to declare a `touches:` set naming a **domain module** (e.g. `server-module/src/battle.rs`) rather than the whole `lib.rs`, so two server slices touching different domains are `touches:`-disjoint and may fan out per `PLAN.md` §9.

**Ownership-guard consolidation (no behavior change)**
- THE repeated `owner != ctx.sender` rejection preamble SHALL be expressed once as `require_owner(ctx, reducer, owner) -> Result<(), String>` (or equivalent), and every reducer SHALL reject an unauthorized `ctx.sender` with the **same** `Err` + `log_reject` output as before. PROOF-OF-TEETH: the existing reducer-security evals/tests (`battle-reducer-security`, `recruit-reducer-security`) SHALL still pass and SHALL still fail their known-bad fixtures.

**Spike (gating; verify-first per `validation-checklist.md`)**
- BEFORE the bulk move, a spike SHALL move exactly one `#[table]` and one `#[reducer]` into a submodule, run `just build` + `just gen` + `just eval`, and confirm zero bindings/schema diff and a green build — establishing that submodule registration works on `spacetime 2.6.0`. IF it does not, the milestone falls back (§6) rather than proceeding blind.

**Content glob loading ((B); ADR-0056)**
- WHEN `game-core` is built THE SYSTEM SHALL embed every `content/<registry>/*.ron` for each glob-loaded registry via a `build.rs` in **sorted filename order**, and the loader SHALL concatenate the parsed `Vec`s — with **no `content.rs` edit required to add a new content file** (the fan-out property).
- THE migrated content SHALL be **row-identical** to pre-change: `load_species()`/`load_skills()`/… return the same rows in the same order; `validate_content` passes; the `append-only-ids` eval is unchanged. PROOF-OF-TEETH: a content-parity test asserts the merged registry equals the pre-migration registry, and a deliberately **malformed** fixture `*.ron` in a registry dir is **rejected** loudly at build/`validate_content` (parse-don't-validate preserved).
- THE loader SHALL remain **pure/deterministic** (compile-time embed + pure parse; no runtime I/O, no wall-clock/RNG — the determinism gate is unaffected), and SHALL change **neither `module_bindings` nor the schema snapshot** (content is data, not schema).

## 4. Plan (slices + `touches:` for collision-safe fan-out)

**Workstream (A)** has two phases: a **serial spine** (the *move* — every extraction touches `lib.rs`, so the fan-out model forces it serial) and a **parallel tail** (per-domain *test extraction* — each touches only one module + its new sibling file, so they are pairwise-disjoint and fan out). Recognizing the split is the point: the move can't parallelize (one shared file), but it is a one-time cost; the tail then **demonstrates the very fan-out this milestone unlocks**. **Workstream (B)** (content, below) is a single slice with disjoint `touches:` that runs **concurrently with all of (A)**.

**The module map below is the canonical `touches:` vocabulary.** ADR-0055 fixes these module boundaries so every downstream milestone (M9, M10, …) declares a *domain module* in its `touches:` set (`server-module/src/battle.rs`) instead of the whole `lib.rs` — which is what makes two server-side slices disjoint. Keep the file names stable once ADR-0055 lands; renaming a module later invalidates downstream `touches:` declarations.

Proposed target layout (the builder may refine — the boundary, not the exact file names, is what ADR-0055 fixes):
```
server-module/src/
├─ lib.rs          # mod wiring + init / sync_content / on_disconnect
├─ schema.rs       # the 15 #[table] structs + row types (lib.rs:33–284)
├─ guards.rs       # log_reject, validate_name, authorize_move, check_party_size,
│                  #   check_monster_in_party, check_team_coupling, require_owner (NEW)
├─ marshal.rs      # char_state, apply_state, monster_from_instance, pub_from_monster,
│                  #   battle_monster_from_row, write_back_hp, encounter_rows_from_table,
│                  #   table_from_encounter_row, skill_defs_from_rows, type_chart_from_rows,
│                  #   loser_base_stat_total, wild_battle_monster
├─ content.rs      # sync_content_inner + seeding helpers
├─ movement.rs     # join_game, enqueue_move, set_move, clear_queue, movement_tick
├─ monster_mgmt.rs # set_nickname, set_party_slot
├─ battle.rs       # start_battle, start_wild_battle, submit_attack, swap_active, flee,
│                  #   heal_party + begin_encounter, lead_party, write_back_party_hp,
│                  #   write_back_battle_results   (the ~900-line cluster)
└─ taming.rs       # attempt_recruit, grant_bait, grant_item, consume_one
```

**Serial spine (all touch `lib.rs` — cannot fan out):**
- **M8.9a — Spike + scaffold + ADR** · `touches: server-module/src/lib.rs, server-module/src/schema.rs (new), docs/adr/0055-* (confirm number)` — move ONE table + ONE reducer to a submodule, prove zero bindings/schema diff + green build; create the empty module files + `mod` declarations; write ADR-0055. **Gating** — if the spike fails, stop and invoke the §6 fallback.
- **M8.9b — The move** · `touches: server-module/src/{lib,schema,guards,marshal,content,movement,monster_mgmt,battle,taming}.rs` — relocate each domain's production code **and its inline `#[cfg(test)] mod tests`** out of `lib.rs` into its module (so `lib.rs` ends as wiring + lifecycle, and *each module owns its own inline tests*), and add `require_owner`, routing every reducer through it. **Recommended as one atomic, gate-proven slice** — a single byte-identical-bindings diff is the cleanest possible proof of no behavior change. (If smaller diffs are preferred, a tight serial chain 9b1 schema+guards+marshal → 9b2 movement+monster_mgmt+content → 9b3 battle+taming is allowed, but it adds no parallelism — see §6.)

**Parallel tail (pairwise-disjoint `touches:` → fan out, bounded N≤2 per `docs/routing.md`):** after 9b each module owns its inline tests, so extracting a module's tests to a sibling file touches **only that one module + its new test file** — so any two of these run concurrently:
- **M8.9c-battle** · `touches: server-module/src/battle.rs, server-module/src/battle_tests.rs` — `battle.rs`'s inline tests → `battle_tests.rs`.
- **M8.9c-taming** · `touches: server-module/src/taming.rs, server-module/src/taming_tests.rs`.
- **M8.9c-movement** · `touches: server-module/src/movement.rs, server-module/src/movement_tests.rs`.
- **M8.9c-marshal** · `touches: server-module/src/marshal.rs, server-module/src/marshal_tests.rs` — (and `content`/`monster_mgmt` likewise *iff* their inline test modules are large enough to be worth a sibling; small ones stay inline — no slice).
- Each declares `#[cfg(test)] mod <domain>_tests;` and moves the bodies verbatim (assertions unchanged). Disjoint by construction → the supervisor may run any two at once.

**Workstream (B) — content glob loading (runs in parallel with ALL of workstream A — disjoint `touches:`):**
- **M8.9e — Content directories + `build.rs`** · `touches: game-core/build.rs (new), game-core/src/content.rs, game-core/content/**, game-core/Cargo.toml (build-deps only), evals/append-only-ids.eval.mjs (only if content paths change)` — add the `build.rs` glob; migrate each registry to `content/<registry>/*.ron` (identical rows/ids); concatenate the parsed `Vec`s in the loaders; content-parity + malformed-fixture proof-of-teeth; ADR-0056. **Disjoint from A** (`game-core/content` + loader vs `server-module/**`) → runs concurrently with the server spine/tail. **Serial only against M8.8's 8c** (which also edits `content.rs` for `skill.accuracy`) → rebase after it; independent of all of A.

- **M8.9d — doc-keeper** · `touches: ARCHITECTURE.md, CHANGELOG.md, memory` — record the module map **and** the content-directory layout (both `touches:` vocabularies); verifier confirms bindings-drift = 0 + schema-snapshot unchanged + content-parity across the whole milestone.

Recommended order: **A:** 9a → 9b (serial spine) → { 9c-battle ‖ 9c-taming ‖ 9c-movement ‖ 9c-marshal …, N≤2 }; **B:** 9e runs **concurrently with A** (disjoint; rebased after M8.8 8c); **then 9d** once both close. The two workstreams are themselves M8.9's demonstration of `touches:`-disjoint fan-out — the server-code reorg and the content-layout change proceed in parallel.

- **DECISION — gating spike outcome / fallback.** If `#[table]`/`#[reducer]` do **not** register from a submodule on `spacetime 2.6.0` (low probability — registration is inventory-based, not path-based), fall back to a **lighter** split: keep `#[table]`/`#[reducer]` *definitions* in `lib.rs` but move all helper fns + the inner logic (`*_inner`) into domain modules that `lib.rs` calls. This still cuts `lib.rs` substantially and still narrows `touches:` for helper-heavy work, just less than a full split. **Your call** is pre-authorized to the spike: proceed with the full split iff the spike is green, else take the fallback. Record which in ADR-0055.
- **DECISION — atomic move vs. serial-chained move (9b).** Slicing the *move* finer (9b1/9b2/9b3) buys **no parallelism** — every move-slice touches `lib.rs`, so they run serial either way. The only trade is diff size vs. a single byte-identical-bindings proof. Default: **one atomic 9b** (cleanest proof; the parallelism lives in the 9c-* tail, not the move). The builder may chain 9b1→9b3 if smaller diffs are easier to verify. **Your call.** Note: the *parallelism* of this milestone is entirely in the 9c-* test-extraction tail (pairwise-disjoint), so do not expect the move to fan out.
- **DECISION — `game-core` inline-test extraction.** Worthwhile for consistency but lower value and it collides with M8.7a/M8.8 combat-test churn. Default: **defer** to a later housekeeping slice; do not bundle into M8.9. **Your call** if you'd rather fold it in once combat work settles.
- **DECISION — content load mechanism (B).** (a) **`build.rs` codegen (recommended)** — a host build script emits the per-registry `include_str!` list from `content/<registry>/*.ron`; **no runtime dependency**, keeps the pure determinism-critical core dep-free (~20 lines). (b) **`include_dir` crate** — simpler code but **adds a runtime dependency** to `game-core` (weigh against (a) for the pure core). (c) **defer to M11** — keep single files until the authored world actually explodes content. **Recommendation: (a)** now (M9/M10 add content imminently; `build.rs` avoids a core dependency). Record the choice in ADR-0056. **Your call.**
- **DECISION — content directory grain.** Per-**registry** dirs now (`content/species/*.ron`) is the simplest fan-out win. Per-**zone**/domain grouping (`content/zones/<zone>/{species,encounters}.ron`) is better once M11 authors multiple zones. **Recommendation: per-registry now; revisit at M11.**
- **Content migration safety (B).** Each registry file is a complete RON `[ … ]` list; the loader concatenates the **parsed `Vec`s** (not strings) in **sorted filename order**, so the merged registry is row-identical and order-stable. Migration preserves every `id` (the `append-only-ids` eval enforces this) and keeps `validate_content` green; the **content-parity** proof-of-teeth (merged == pre-migration) is the gate. `type_chart` may stay a single file.
- **Excluded from M8.9 (see §2), tracked elsewhere:** integration-runtime isolation (harness/test-infra + supervisor resource rules), shared-type pre-splitting (the extending milestone's job — YAGNI), and `sccache`-by-default (harness build-infra). Named here so they are not silently dropped.
- **ADR numbering (re-confirm at build time).** Highest filed at `92114af` is `0053`; M8.6c/6d/M8.7/M8.8 consume numbers continuously — confirm next-free in `docs/adr/` and shift up if taken.
- **Macro hygiene during the move.** Reducers must stay `pub`; `use` paths for `game_core::*` and `spacetimedb::*` re-imported per module; the `movement_tick_schedule` scheduled-table + its `movement_tick` reducer should live together (`movement.rs`) so the `scheduled(movement_tick)` reference resolves. The schema-snapshot + `clippy -D warnings` + `cargo check --all-targets` catch any slip.

### Red-team
- A reorg milestone that changes *anything* observable is a bug, not a refactor. The bindings-drift + schema-snapshot gates are non-negotiable proof-of-no-change; if either moves, the slice is wrong. This is the whole reason the split is safe to do on a live, fleet-built project.
- The benefit is **parallelism**, and that benefit is only realized if future slices actually declare domain-scoped `touches:` sets. ADR-0055 should state the module map *as the canonical `touches:` vocabulary* so later specs name `server-module/src/battle.rs`, not the file that no longer holds battle.
- Doing this *before* M8.7/M8.8 would re-serialize the very work it aims to speed up. Sequence discipline is part of the design.

### Simplify
No new subsystems, no schema, no rules, no game-design surface, no new crate, no runtime dependency. Workstream (A) is file moves, a `mod` declaration, one consolidating helper (`require_owner`), and a test relocation. Workstream (B) is one host `build.rs` + a content-file migration (data only). The SSOT/parity/determinism/privacy-split spine is untouched; the only thing that changes is *where the code and content live*, proven by identical bindings/schema (A) and content-parity (B).

> **# ADR-0055: server-module internal module boundary**
>
> **Status:** Proposed → Accepted (M8.9) · **Date:** (build date) · **Deciders:** Drew
>
> **Context.** `server-module/src/lib.rs` grew to ~2081 production lines (15 tables, 18 reducers, 27 helpers) in a single namespace. Under the fleet's `touches:`-disjoint parallel-build model (`PLAN.md` §9, `WORKSPACE-PLAN.md` §7), any slice touching `lib.rs` serializes against every other — so all server-side gameplay work is forced serial. The file also taxes review and impact analysis. `standards/adr-process.md` requires an ADR to introduce a new module boundary.
>
> **Decision.** Split the crate into cohesive domain submodules — `schema`, `guards`, `marshal`, `content`, `movement`, `monster_mgmt`, `battle`, `taming` — with `lib.rs` reduced to module wiring + lifecycle reducers (`init`/`sync_content`/`on_disconnect`). Consolidate the per-reducer ownership-check preamble into `guards::require_owner`. Extract inline `#[cfg(test)]` modules to sibling `*_tests.rs` files (matching the existing game-core convention). This module map is the canonical `touches:` vocabulary for future server-side slices.
>
> **Consequences.** (+) Server slices touching different domains become `touches:`-disjoint and parallelizable; smaller diffs; tighter blast radius. (+) Behavior is provably unchanged — table/reducer names are explicit, so regenerated bindings and the schema snapshot are byte-identical (the acceptance gate). (−) A one-time serial reorg (M8.9's own slices share `lib.rs`). (−) A modest learning cost to know which module owns a reducer (mitigated by this ADR's map). **Considered alternatives:** (a) leave as one file — rejected: the parallelism bottleneck persists and grows with M11–M16. (b) Split `game-core`/`server-module` into more *crates* — rejected as YAGNI; intra-crate modules suffice and keep the workspace simple (ADR-0005). (c) Lighter "logic-in-modules, macros-in-lib.rs" split — retained as the **fallback** if the submodule-registration spike (M8.9a) fails on `spacetime 2.6.0`.
