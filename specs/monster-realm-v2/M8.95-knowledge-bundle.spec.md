# Spec: M8.95 — Agent Knowledge Bundle (OKF-aligned, generated)

**Status:** draft · **Owner:** Drew · **Date:** 2026-06-27
**Stack:** spacetimedb-game · **Project:** monster-realm · **Sequenced between M8.9 and M9.**
**Depends on:** **M8.9** (the producer parses the post-M8.9 `server-module/src/schema.rs` + the domain-module reducer layout; running before M8.9 would target the soon-to-be-moved `lib.rs`). Independent of M9.
**Decisions:** harness **ADR-0008** (knowledge contract) + `standards/knowledge-format.md`; project **ADR-0057** (generated knowledge bundle — corpus copy filed; the project-side implementation ADR is mirrored into `docs/adr/` at the confirmed next-free number at build — M8.9 consumes 0055/0056 first, **confirm before creating**). Upholds **ADR-0006** (additive schema — this milestone changes **no** schema), **0010** (proof-of-teeth), **0040** (privacy split — surfaced in the bundle), **0007/0042/0044/0045/0046** (the public/private table posture the bundle records). Reuses the **ADR-0050** drift-in-CI pattern.
**Workflow:** sourced from `OKF-research-and-impact-analysis.md` (2026-06-27 OKF deep-dive + harness/monster-realm fit analysis). **Pure additive tooling milestone — no schema, no rule, no game-design surface.** The gate is "the bundle regenerates byte-identically from source and conforms; proof-of-teeth fixtures are rejected."

## 1. Problem / intent

Agents (and humans) working in monster-realm most need to understand the **SpacetimeDB schema and reducer surface**: which tables exist, their columns/PKs, which are **public vs. private** and why (the ADR-0040 privacy split, the wild-gene/encounter privacy of ADR-0044/0045), and how rows relate (FKs, projections). That knowledge is well-kept today but **scattered** across `server-module/src/` (post-M8.9: `schema.rs` + domain modules), the 34 KB `ARCHITECTURE.md`, ~20 ADRs, and the schema-snapshot baseline — there is no single, portable, agent-navigable surface. The schema is precisely the artifact OKF was built to catalog.

Per harness **ADR-0008**, we adopt a thin **knowledge contract** (`standards/knowledge-format.md`) — a deliberately small subset of Google's OKF — rather than OKF wholesale. This milestone realizes that contract for monster-realm: a **generated, drift-checked, OKF-conformant bundle** projecting the schema/reducer surface, plus bringing the existing research library into conformance. The hard constraint, from AGENTS.md rule #3, is **SSOT: the bundle is a generated *projection*, never a second hand-maintained copy** — drift fails CI. Success = `just eval` regenerates the bundle byte-identically from source, every concept conforms (required frontmatter, `type ∈ vocabulary`, links resolve), and a stale/malformed fixture is rejected.

## 2. Scope

**In scope** (the curated OKF subset adopted per ADR-0008; criteria §3, slices §4)
- **Producer (generate from source).** `scripts/okf-export.mjs` walks `server-module/src/schema.rs` by **reusing the already-exported `parseTableSchemas()`** from `evals/battle-schema-snapshot.eval.mjs` (the same parser that gates schema drift now *feeds* the bundle → they cannot disagree), plus a reducer-signature pass over the domain modules and ADR cross-references, and emits the bundle. The producer is the **sole writer** of generated concepts.
- **The bundle** at `docs/knowledge/` — committed as a generated artifact (like `module_bindings/` and `evals/baselines/table-schemas.json`), one concept per table (`type: SpacetimeDB Table`: columns, PK, `visibility: public|private`, `resource:` → `server-module/src/schema.rs#Lstart`, bundle-relative links to related tables, e.g. `monster` → `monster_pub` projection, `battle` participants) + one concept per reducer (`type: SpacetimeDB Reducer`: signature, the table(s) it writes, ownership/guard note) + a `Schema Overview` concept + a generated `index.md` (progressive disclosure).
- **Conformance + drift gate.** `evals/knowledge-bundle-conformance.eval.mjs` (runs under `just eval` → `just ci`): (a) **lint** every concept via the vendored `okf-lint` (required frontmatter, `slug == path`, `type ∈ registered vocabulary`, abstract single-line ≤120, every bundle-relative link resolves); (b) **drift** — re-run the producer in `--check` mode and FAIL if the committed bundle differs from a fresh generation. **Proof-of-teeth:** a malformed concept fixture AND a stale-bundle fixture must both be flagged (ADR-0010).
- **Vendored linter.** A project-local copy of the harness-canonical `okf-lint.mjs` (the research-index precedent: harness owns `scripts/okf-lint.mjs`, the project vendors it — *no cross-repo path reaching*, per the WORKSPACE-PLAN anti-coupling rule). Lands at `.claude/hooks/okf-lint.mjs` alongside the existing vendored `research-index.mjs`.
- **Research-library conformance.** Add `type: Research Note` (additive) to each `docs/research/*.md`; extend the vendored `research-index.mjs`/`research-lint.mjs` to carry/validate `type` so the research library is the second conformant instance of the same contract.
- **Wiring.** `just knowledge` (regenerate the bundle) and `just knowledge-check` (drift `--check`); the conformance eval is auto-discovered by `evals/run.mjs`. Optionally wire bundle regeneration into the existing `format-edited` write hook (as `research-index` already is) so a schema edit refreshes the bundle locally.

**Harness-side prerequisites** (separate repo; governed by ADR-0008 — landed alongside this spec, not built in the monster-realm worktree)
- `standards/knowledge-format.md` (**already authored**) — the contract SSOT.
- The harness-canonical `scripts/okf-lint.mjs` and `type`-aware `research-index.mjs`/`research-lint.mjs`, plus their vendoring into `_base`/`spacetimedb-game` templates so new projects inherit the contract. (Mechanically these are harness changes; this spec consumes them.)

**Out of scope (named deferrals — declared, not dropped)**
- **Hand-authoring any generated concept.** The producer is the only writer; a human edit to `docs/knowledge/**` is a drift-gate FAIL by design (SSOT).
- **Google's reference producer (Gemini/BigQuery) and HTML visualizer as a CI dependency.** Wrong source system / non-deterministic / the reference viewer has a known link-handling bug. The static visualizer MAY be used ad-hoc for human browsing (feed it relative, untitled links), never as a gate.
- **Non-schema concept types** beyond `Schema Overview` + research (e.g. `Runbook`, `Metric`) → add when a real consumer needs them; the vocabulary is in `standards/knowledge-format.md` and extends one line at a time. YAGNI now.
- **`log.md` per-concept history** → redundant with git + the Conventional-Commits `CHANGELOG.md` + ADRs; not adopted (do not duplicate history).
- **RDF/typed-relationship graph, managed/graph memory backend, a knowledge *service* or MCP endpoint** → the contract leaves room; nothing requires them now (each is its own future ADR).
- **Ingesting any third-party OKF bundle** → OKF v0.1 has no provenance/signing; an external bundle is untrusted input (AGENTS.md rule #6). The bundle derives only from in-repo source.
- **Schema/rule/game-design change of any kind.** If a diff changes the schema snapshot or generated bindings, it is a bug in this milestone, not an intended change.

## 3. Acceptance criteria (EARS)

**Producer / SSOT (the spine)**
- WHEN `scripts/okf-export.mjs` runs THE SYSTEM SHALL emit one `SpacetimeDB Table` concept per `#[spacetimedb::table]` in `server-module/src/schema.rs` — derived via the exported `parseTableSchemas()` — with columns, PK, and `visibility` (public iff the table is declared `public`), and SHALL emit one `SpacetimeDB Reducer` concept per `#[reducer]` across the domain modules.
- THE producer SHALL set each generated concept's `updated`/`source` from the source (commit or file mtime + path), and SHALL be **deterministic** (same source → byte-identical bundle; sorted ordering), so the drift gate is stable.
- THE producer SHALL be the **only** writer of `docs/knowledge/**`; IF a generated concept is edited by hand THEN the drift gate SHALL FAIL on the next `just eval`.
- THE bundle SHALL change **neither** the schema snapshot **nor** `module_bindings` (it reads them; it is schema-neutral).

**Conformance lint**
- WHEN the conformance eval runs THE SYSTEM SHALL FAIL if any concept is missing a required frontmatter key (`type`, `title`, `slug`, `updated`, `tags`, `abstract`), has `slug != path`, has a `type` **not** in the registered vocabulary (`standards/knowledge-format.md`), has a multi-line abstract, or contains a bundle-relative link that does not resolve to a file in the bundle.
- THE consumer/lint SHALL **tolerate** unknown frontmatter keys (forward-compat) and SHALL NOT fail on a recognized-but-extra key.

**Drift gate (reuses the ADR-0050 pattern)**
- WHEN the conformance eval runs in CI THE SYSTEM SHALL regenerate the bundle to a temp location and FAIL with a readable diff IF it differs from the committed `docs/knowledge/**`.
- PROOF-OF-TEETH: the eval SHALL include (a) a **malformed** concept fixture (e.g. missing `type`, or a dangling link) that the lint MUST flag, and (b) a **stale-bundle** fixture (a committed concept whose source has "changed") that the drift check MUST flag — and the eval SHALL FAIL if either tooth does not bite.

**Research-library conformance**
- WHEN `research-lint` runs after this milestone THE SYSTEM SHALL require `type` on every `docs/research/*.md` and accept `type: Research Note`; the existing `confidence`/`sources`/`status`/`supersedes` fields SHALL remain required/valid (the contract is a **superset** — no field is dropped).
- THE research `INDEX.md` SHALL regenerate to include the `type` column and remain in sync (the existing `--check` gate passes).

**Privacy posture surfaced (no behavior change)**
- THE bundle SHALL mark `monster`, `encounter`, and `battle_wild` as `visibility: private` and `monster_pub`/`battle`/`inventory` as `visibility: public`, matching ADR-0040/0042/0044/0045/0046; a private table's concept SHALL link to its public projection where one exists (`monster` → `monster_pub`). (This makes the privacy split machine-checkable knowledge, not just prose.)

**Wiring**
- THE conformance eval SHALL be auto-discovered by `evals/run.mjs` (named `*.eval.mjs`, default export → `{name, pass, detail}`) and run under `just eval`/`just ci`; `just knowledge` SHALL regenerate the bundle and `just knowledge-check` SHALL run the drift `--check`.

## 4. Plan (slices + `touches:` for collision-safe fan-out)

Design anchors: the producer reuses the **exported** `parseTableSchemas()` (no new parser, no second schema truth); the bundle is committed + drift-checked exactly like `module_bindings`/`table-schemas.json`; the vendored `okf-lint` follows the `research-index.mjs` precedent (harness-canonical → project-vendored, **never** a cross-repo path import). All `touches:` targets assume the **post-M8.9** tree (`server-module/src/schema.rs` + domain modules).

Dependency chain: the producer (8.95a) gates the eval (8.95b, consumes the bundle it emits). The research-conformance slice (8.95c) is disjoint (`docs/research/**` + vendored research scripts) and fans out against 8.95a/b. doc-keeper (8.95d) closes.

- **M8.95a — Producer + bundle** · `touches: scripts/okf-export.mjs (new), docs/knowledge/** (new, generated), justfile (knowledge / knowledge-check verbs), .claude/hooks/okf-lint.mjs (vendored from harness)` — implement `okf-export.mjs` (import `parseTableSchemas` from the schema-snapshot eval; reducer pass; ADR/visibility cross-ref; deterministic sorted emit; `index.md`); generate and commit the bundle; add `just knowledge`/`knowledge-check`. **Critical-path start.**
- **M8.95b — Conformance + drift eval** · `touches: evals/knowledge-bundle-conformance.eval.mjs (new), evals/baselines/ (only if a fixture file is added)` — lint (via vendored `okf-lint`) + drift (`okf-export --check`) + both proof-of-teeth fixtures. **Serial after 8.95a** (gates the bundle 8.95a produces).
- **M8.95c — Research-library conformance** ‖ (disjoint from a/b) · `touches: docs/research/*.md, .claude/hooks/research-index.mjs, .claude/hooks/research-lint.mjs (vendored, type-aware), docs/research/INDEX.md` — add `type: Research Note` to each research doc; vendor the `type`-aware research-index/lint; regenerate `INDEX.md`. Touches only `docs/research/**` + the vendored research scripts → **fans out with 8.95a/b** (N≤2 per `docs/routing.md`).
- **M8.95d — doc-keeper + verifier** · `touches: ARCHITECTURE.md, CHANGELOG.md, docs/adr/00NN-* (project impl ADR mirror of corpus 0057 — confirm next-free), memory` — record the bundle layout + the `docs/knowledge/` location in ARCHITECTURE.md; file the project-side ADR; CHANGELOG + memory; verifier confirms `just ci` green, schema snapshot + bindings unchanged, and the bundle drift-clean. **Closes the milestone.**

Recommended order: **8.95a → 8.95b → { 8.95c ‖ 8.95d }** (8.95c may also start immediately — it is disjoint from a/b).

**Boundary preview — what later milestones consume:** M9+ schema additions (new tables/reducers) regenerate the bundle automatically via `just knowledge` (no manual concept authoring); a future memory backend or viewer can consume `docs/knowledge/**` unchanged (the portable contract); new concept types (`Runbook`, `Metric`) register one line in `standards/knowledge-format.md` when a consumer needs them.

## 6. Risks / decisions
- **Sequence after M8.9 (hard).** The producer parses `server-module/src/schema.rs` and the per-domain reducer modules — the layout M8.9 creates. If M8.9 has **not** landed, the producer would have to parse the `lib.rs` monolith and then break when M8.9 moves it. Gate M8.95 on M8.9 closing. (Also: M8.9 updates `battle-schema-snapshot.eval.mjs`'s `SERVER_SRC` to `schema.rs`; the producer imports `parseTableSchemas` from that same module, so it follows automatically.)
- **DECISION — reuse `parseTableSchemas` vs. a new parser.** Reuse (recommended): the gate parser and the bundle producer share one code path, so a parser bug can't let the bundle and the schema snapshot disagree. A second parser would be a second schema truth (SSOT violation). **Your call** is pre-authorized to reuse; only fork if the reducer pass needs materially different parsing (then export a shared helper from one module, don't copy).
- **DECISION — bundle committed vs. gitignored.** Commit it (recommended), like `module_bindings/` and `table-schemas.json`: a committed bundle is diff-reviewable and the drift gate has something to compare against. Gitignoring it loses both. **Your call.**
- **DECISION — reducer concepts now, or tables-only v1.** Recommended: tables + a `Schema Overview` first (highest value, smallest surface), reducers in the same slice if cheap, else a fast-follow. Reducers are read from domain modules (more parsing than tables). **Your call** — tables-only is a valid v1 that still satisfies the core intent; add reducers when the parsing is proven.
- **DECISION — bundle path `docs/knowledge/` vs. `knowledge/`.** `docs/knowledge/` (recommended) keeps it beside `docs/research/` and `docs/adr/` (all agent knowledge under `docs/`). **Your call.**
- **OKF v0.1 churn.** OKF is an unfinished draft and may make breaking changes. Mitigation: the adopted surface is deliberately tiny (one required field + link convention) and lives behind our own `standards/knowledge-format.md`, so an upstream change costs a small producer/lint edit, not a migration. We do **not** depend on OKF's tooling.
- **Cross-repo coupling.** The harness owns the canonical `okf-lint.mjs`; the project **vendors a copy** (the `research-index.mjs` precedent). Do **not** `import` across `../../scripts/` — that re-introduces the hidden-coupling anti-pattern (WORKSPACE-PLAN §13). The harness `sync-templates` drift tool keeps vendored copies fresh.
- **Bundle staleness between regenerations.** A schema edit not followed by `just knowledge` makes the committed bundle stale — caught by the drift gate in CI (fails the PR), and optionally by wiring regeneration into the `format-edited` hook for local freshness. Staleness can never reach `main` green.
- **Privacy leakage via the bundle.** The bundle records *that* a table is private and *why* (ADR ref) — it MUST NOT reproduce private row data (it describes schema, not rows). The producer reads source + ADRs only; it never queries a live DB. (Confirms ADR-0040's posture is documented, not exposed.)

## 7. Review / red-team / simplify notes
### Source
`OKF-research-and-impact-analysis.md` (2026-06-27): OKF deep-dive + harness/monster-realm fit. Conclusion adopted here: take OKF's *conventions* (type field, link graph, index) as a generated, CI-gated projection; reject wholesale adoption (SSOT/drift, v0.1 immaturity, single-vendor gravity, no provenance). Recorded as harness ADR-0008 + project ADR-0057.

### Red-team
- **A bundle that drifts from the schema is worse than no bundle** (it misleads agents with confident-but-stale facts). The drift gate is therefore non-negotiable and ships proof-of-teeth — if the gate doesn't bite, the slice is wrong (ADR-0010).
- **A hand-edit to a generated concept** is the most likely SSOT regression. Mitigation: the producer is the sole writer; any hand edit fails the drift gate; the concept files carry `source:` so a reader sees they are derived.
- **OKF's no-provenance weakness** is contained by only ever generating from in-repo source and never ingesting external bundles (AGENTS.md rule #6).

### Simplify
No new subsystem, schema, rule, or game-design surface. The producer **reuses** the existing schema parser; the gate **reuses** the ADR-0050 drift pattern and the `evals/run.mjs` harness; the linter **reuses** the `research-index` vendoring precedent; the research library only **gains a field**. The whole milestone is "generate a projection of truth we already have, and fail CI if it drifts." If even this is more than wanted, the **minimum viable** subset is 8.95a (tables-only bundle) + 8.95b (drift gate) — research conformance (8.95c) can be a later housekeeping slice.
