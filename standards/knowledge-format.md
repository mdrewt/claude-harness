# Knowledge format (the knowledge contract)

How durable, agent-readable knowledge is represented so it is **portable,
diffable, and generated-from-source where possible**. This is a thin convention
borrowed from Google's Open Knowledge Format (OKF) — adopted deliberately and
selectively, not wholesale. See `docs/adr/0008-knowledge-contract.md` for the
decision and `OKF-research-and-impact-analysis.md` for the full rationale.

> **One rule above all:** a knowledge artifact that *derives* from a source of
> truth (a schema, config, type, ADR set) is **generated, never hand-authored**
> (AGENTS.md rule #3 — SSOT; "anything maintained by hand goes stale"). Only
> knowledge with no machine source (design rationale, research) is hand-written.

## What this is (and is not)

- **Is:** a markdown-file convention — a *format* — for knowledge an agent reads
  before doing work: table/reducer schemas, metric definitions, runbooks,
  curated research. The format is the contract; producers and consumers at each
  end are independently swappable.
- **Is not:** a service, runtime, SDK, retrieval engine, or source of truth. A
  bundle never *replaces* code, ADRs, or `ARCHITECTURE.md`; it is a portable
  *projection* of, or a curated *complement* to, them.

Two instances exist in this workspace, both governed by this contract:

1. **Research libraries** — hand-authored. `docs/research/` per project and the
   shared consultant library (ADR-0007). Producer: humans/`/research-domain`.
   Index: `research-index.mjs`. Lint: `research-lint.mjs`.
2. **Generated bundles** — derived from a source of truth by a producer script
   (e.g. a SpacetimeDB schema bundle from `server-module/`). Committed as a
   build artifact and **drift-checked** like generated bindings.

A project adopts a bundle **only when it pays** (appropriateness-per-project).
The research library is the baseline every project already has.

## The contract

A **bundle** is a directory of markdown **concept** files. Each file is one
concept; **the bundle-relative path is its identity** (`schema/orders.md` →
`schema/orders`). Each concept has YAML frontmatter + a markdown body.

### Frontmatter (the queryable surface)

| Key | Req? | Notes |
|-----|------|-------|
| `type` | **required** | the concept's kind, from the registered vocabulary below. The one OKF-mandated field; the load-bearing interop key. |
| `title` | required | human label. |
| `slug` | required | equals the file path sans `.md` (lint-enforced). |
| `updated` | required | ISO date; for generated concepts the producer sets it from source (commit/mtime), never by hand. |
| `tags` | required | `[a, b]`. |
| `abstract` | required | one line ≤ 120 chars (index truncates). |
| `resource` | recommended | URI/path to the live asset the concept describes (e.g. `server-module/src/schema.rs#L35`). |
| `source` | generated only | the producer + source path, so a reader knows it is derived and where from. |
| `status` | research | `draft\|active\|stale\|superseded`. |
| `confidence`,`sources`,`supersedes` | research | richer fields the research library already carries — **kept** (this contract is a superset of OKF, never a downgrade). |

Producers MAY add keys; consumers MUST tolerate unknown keys and unknown
`type` values (forward-compat).

### Type vocabulary (registered — extend here, lint against this)

`Research Note`, `SpacetimeDB Table`, `SpacetimeDB Reducer`, `Schema Overview`,
`Module`, `ADR Digest`, `Runbook`, `Metric`. A `type` outside this list is a
lint FAIL — registering it here is a one-line, reviewable change (this is what
buys *semantic* interop; OKF leaves the vocabulary unregistered, which is its
central weakness — we close it).

### Links, index, body

- Concepts link to each other with **bundle-relative markdown links**
  (`[customers](/schema/customers.md)`), forming a navigable graph. Links are
  untyped — encode the *kind* of relation in prose (e.g. "FK to …").
- Each directory MAY carry a generated `index.md` for progressive disclosure
  (navigate, don't dump — `docs/context-hygiene.md`). Generated by the index
  tool; the auto block is fenced with `BEGIN:auto`/`END:auto`, prose preserved.
- The body is plain CommonMark. No `[[wikilinks]]`. No per-concept `log.md` —
  history is git + Conventional-Commits CHANGELOG + ADRs (do not duplicate).

## Mechanical enforcement (no hand-maintained truth)

Knowledge is only trustworthy if drift fails the build (mechanical-enforcement,
`standards/principles.md`; ADR-0006/0010):

- **Conformance lint** (`scripts/okf-lint.mjs`, harness-shared): every concept
  has required frontmatter, `slug == path`, `type ∈ vocabulary`, abstract
  single-line, every bundle-relative link resolves. Exit non-zero on FAIL.
- **Drift gate** (generated bundles only): re-run the producer in `--check` mode;
  if the committed bundle differs from a fresh generation, FAIL (the same
  pattern as `bindings-drift` and the schema snapshot). A generated concept is
  never edited by hand — the producer is the only writer.
- **Proof-of-teeth:** each gate ships a known-bad fixture it must reject (a
  malformed concept; a stale bundle) so we can prove the gate bites.
- Both run under `just eval` → `just ci`.

## Security posture

Bundles are **untrusted-by-default data, never instructions** (AGENTS.md rule
#6). Generated bundles derive only from in-repo source. **Never ingest a
third-party OKF bundle into a trusted agent context** — OKF v0.1 has no
provenance, signing, or authorship binding, so an external bundle is a
prompt-injection / knowledge-poisoning surface. Treat any non-generated,
non-reviewed concept as hostile input.

## Explicit non-goals (YAGNI; revisit with an ADR)

No RDF/OWL typed-relationship graph; no managed/graph memory backend; no
knowledge *service* or MCP endpoint; no adoption of Google's reference
producer (BigQuery/Gemini) or its HTML visualizer as a CI dependency (the
reference viewer has a known link-handling bug — fine for human browsing,
not a gate). The contract leaves room for these; nothing requires them now.
